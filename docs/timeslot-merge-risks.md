# 타임슬롯 통합(예: 10-11 + 11-12 → 10-12) 검토 문서

상태: **검토 중 — 미결정** (정원 정책 / 과거 이력 범위 결정 필요)
작성일: 2026-06-18

## 배경

운영 중 동일 인원이 `10-11`, `11-12` 두 슬롯에 연속으로 배정되는 경우가 많아,
매번 시간대별로 따로 배정하는 게 번거로워서 `10-12` 단일 슬롯으로 합치는 타임슬롯 설정 변경을 고려 중.

현재 타임슬롯 삭제/추가 시 동작 (`AdminPage.tsx:603-625`):
- 삭제될 슬롯에 기존 배정이 있으면 경고만 띄움 ("DB에 남지만 화면에는 안 보임")
- 자동 마이그레이션/병합 로직 없음 — 기존 배정은 고아 데이터로 남고, 새 슬롯은 빈 상태로 시작

"기존 배정을 새 슬롯으로 자동 통합" 기능을 추가하면 가능한지, 그리고 어떤 위험이 있는지 검토한 내용.

---

## 1. 정원(capacity) 의미가 바뀌는 문제 — 가장 중요한 위험

`cellState.ts:82`
```ts
isFull: dayAssignments.length >= maxCapacity
```

정원은 **슬롯 안의 배정 행 개수**로만 계산되고, `time_sub`(전반/후반 구분)는 전혀 고려하지 않는다.

- 지금: `10-11` 정원 2명 + `11-12` 정원 2명 → 시간대별로 최대 2명씩, 한 시간대당 최대 4명까지 다른 사람이 거쳐갈 수 있음.
- 병합 후 `10-12` 정원을 그대로 "2"로 두면 → 2시간 전체에 **딱 2명까지만** 배정 가능해짐. 한 사람이 10~11만, 다른 사람이 11~12만 담당해도 `time_sub`로 반쪽씩 나눠도 카운트는 똑같이 1씩 차감되기 때문.
- "동일인이 양쪽에 고정 배정"되는 케이스(이번 변경의 주 목적)는 문제 없음 — 한 사람이 풀타임으로 1개 행만 차지.
- 하지만 간헐적으로 시간을 나눠 다른 사람이 들어가는 경우가 조금이라도 있다면, "시간당 N명" 정책을 "2시간당 N명"으로 정확히 동치 변환할 방법이 없음. 정원 숫자를 운영 의도에 맞게 새로 정해야 한다.

## 2. 기존 데이터 병합 시 경우의 수 (이름 기준 4가지 패턴)

| 패턴 | 처리 방향 |
|---|---|
| **A. 동일인이 양쪽에 모두 배정** (주 목적 케이스) | 한 행으로 합치고 `time_sub`를 "전체(10~12)" 값으로 설정. 메모/색상/잠금(`is_locked`)/역할(`role_id`)이 두 행 사이에 다르면 어느 쪽 값을 남길지 정책 필요 (예: 메모는 합치기, 잠금은 OR, 역할 불일치 시 수동 검토 대상으로 표시) |
| **B. 한쪽에만 배정 (다른 쪽은 빈 슬롯)** | `time_sub`를 "전반만"(`10`) 또는 "후반만"(`11`)로 설정해서 이행 — 빈 시간이 채워진 것처럼 보이면 안 됨 |
| **C. 서로 다른 사람이 각각 한쪽씩** | 두 행 모두 유지하되 각각 `time_sub`를 전반/후반으로 분리 설정. 이름이 다르므로 unique 제약 충돌은 없음. 다만 1번 정원 문제가 가장 크게 드러나는 케이스 |
| **D. A+C 혼합** (동일인 풀타임 + 추가로 한쪽에만 다른 사람) | 행별로 위 규칙을 따로 적용해야 해서 단순 "그룹화 후 1행"이 아니라 케이스별 분기 필요 |

DB 유니크 인덱스 (`unique_member_assignment`, `reset_db.sql:247-249`):
```sql
CREATE UNIQUE INDEX unique_member_assignment
  ON assignments (tenant_id, year, month, day, time_slot, member_name)
  WHERE member_type != 'admin_note';
```
패턴 A처럼 동일인이 두 슬롯에 모두 있던 경우, 병합 후 두 행을 그대로 두면 `(date, 새 슬롯, 이름)` 조합이 중복되어 이 제약을 위반함 → 반드시 한 행으로 합치고 나머지 하나는 삭제해야 함.

## 3. 그 외 점검 필요한 지점

- **과거 기록까지 건드릴지** — `assignments`는 전체 기간에 걸쳐 누적되는 테이블이고, 슬롯 설정(`tenant.settings.time_slots`)은 시점 구분 없이 테넌트 전체에 적용됨. "오늘 이후만 새 슬롯 적용 + 과거는 그대로 보존"이 구조적으로 안 됨. 이미 끝난 근무 이력까지 통합하면 "그 시점에 실제로 1시간만 일했는지 2시간 다 일했는지" 같은 이력의 정확도가 왜곡될 위험 있음.
  → **결정 필요: 통합 대상을 "오늘 이후 날짜"로 한정할지, 과거 전체 이력까지 변경할지.**
- **슬롯 모양이 정확히 안 맞는 경우** — 새 슬롯이 삭제되는 슬롯들을 정확히 양분하지 않고 부분적으로만 겹치는 경우(예: 기존이 `10-11`/`11-13`인데 새로 `10-12`를 추가) 자동 통합 로직을 적용하면 안 되고, 기존처럼 경고만 띄우고 수동 처리로 빠져야 함.
- **`schedule_rules`/`slot_settings`** — 날짜별이 아니라 테넌트 전체 설정이라 통합 자체는 단순(새 슬롯에 대해 요일별 운영여부·정원을 한 번만 정하면 됨)하지만, 정원 숫자는 위 1번 문제와 직결.
- **`admin_note`(휴관/비고)** — 위 유니크 인덱스가 `WHERE member_type != 'admin_note'`로 애초에 제외되어 있어 이름 충돌 위험은 없음. 비고는 `time_sub`(절대 시간) 기준으로 매칭되는 별도 로직(`ScheduleGrid.tsx`의 `computeSlotGroups`/`parseTimeSub`)이라 슬롯 문자열 변경의 직접 영향은 적어 보이지만, 실제 동작은 구현 전에 한 번 더 확인 필요.
