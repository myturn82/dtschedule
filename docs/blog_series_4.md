# [제 4편] 첫인상은 5분 안에 — 셋업 위자드와 두 곳에서 살아 숨쉬는 커스텀 필드

신규 조직이 가입 직후 마주하는 화면은 서비스의 첫인상을 결정합니다. 텅 빈 관리자 콘솔에 던져놓고 "알아서 설정하세요"라고 하면 절반은 그 자리에서 이탈합니다. 반대로 질문을 너무 많이 던지면 나머지 절반이 지칩니다.

4편은 이 균형을 잡기 위한 **7단계 셋업 위자드**, 그리고 그 마지막 단계에서 등장하는 **커스텀 필드**가 관리자콘솔과 어떻게 이어져 있는지에 관한 이야기입니다.

---

## 1. 실사용 시나리오: 질문 7개로 조직 하나를 완성하기

### 시나리오: 반려동물 위탁 돌봄 서비스

새로 가입한 반려동물 돌봄 업체 대표를 예로 들어보겠습니다. 이 대표는 개발자도, IT에 능숙한 사람도 아닙니다. 그가 마주치는 질문은 순서대로 이렇습니다.

1. "조직을 소개해주세요" — 이름 하나만 입력합니다.
2. "어떻게 운영할 예정인가요?" — 회원공유/회원개별/비회원 중 하나를 카드 형태로 고릅니다.
3. "운영 시간 단위를 정해주세요" — 한 칸이 1시간인지 2시간인지.
4. "역할이 필요한가요?" — 필요 없으면 건너뜁니다.
5. "언제 운영하나요?" — 요일 템플릿 선택.
6. "추가 정보를 수집할까요?" — 배정 시 이름 외에 무엇을 더 받을지.

각 단계는 "왜 이걸 물어보는지"를 한 줄 설명과 함께 보여줍니다. 그 결과 이 대표는 회원개별 모드, 2시간 단위, 역할 없음, 평일만 운영을 5분 만에 완성하고 바로 캘린더 화면을 마주하게 됩니다.

> **📸 [이미지 삽입 구간: 셋업 위자드 진행바 + 2단계 운영 모드 카드 선택 화면]**
> *설명: 상단에 1~7단계 진행바가 보이고, "회원 공유 / 회원 개별 / 비회원" 세 카드 중 하나에 마우스가 올라가 살짝 강조된 화면.*

### 왜 "역할"과 "추가 정보"는 건너뛸 수 있게 했나

위자드를 설계하며 가장 많이 고민한 지점은 "얼마나 많이 물어볼 것인가"였습니다. 역할 구분이 필요 없는 조직(대부분)에게 역할 이름을 강제로 입력하라고 하면, 그 시점에 이탈할 확률이 올라갑니다. 그래서 4·6단계는 항상 "건너뛰기"가 기본값이고, 필요한 조직만 추가로 입력하는 구조로 만들었습니다.

## 2. 관리자 조작 가이드: 커스텀 필드로 우리 조직만의 질문 추가하기

배정 등록 시 이름·연락처 말고 조직마다 꼭 물어봐야 하는 정보가 있습니다. 병원이라면 "증상", 스터디룸이라면 "인원수", 촬영 스튜디오라면 "촬영 컨셉" 같은 것들입니다. 이걸 코드 수정 없이 관리자가 직접 추가할 수 있게 만든 게 커스텀 필드입니다.

> **📸 [이미지 삽입 구간: 셋업 위자드 7단계 - 커스텀 필드 추가 화면]**
> *설명: "필드 이름", "타입(텍스트/숫자/드롭다운/라디오/체크박스/체크박스그룹/전화번호/이미지첨부)" 선택 후 실시간 미리보기가 오른쪽에 뜨는 화면.*

1. 셋업 위자드 마지막 단계 또는 관리자 콘솔 → **커스텀 필드** 탭에서 **+ 필드 추가**를 누릅니다.
2. 필드 이름과 타입을 고릅니다. 8가지 타입 중 텍스트·숫자·전화번호·드롭다운은 **플레이스홀더**(입력 예시 문구)를 추가로 지정할 수 있습니다.
3. 드롭다운·라디오·체크박스그룹 타입은 선택지를 직접 입력합니다. 이때 선택지마다 "단위"(금액·수량·인원·회차)를 지정하면, 나중에 통계 집계 시 자동으로 합산됩니다.
4. **대시보드 노출** 옵션을 켜면, 이 필드의 값이 관리자 대시보드 통계에도 집계됩니다.
5. 저장하면 이후 모든 배정 등록 팝업에 이 필드가 자동으로 추가됩니다.

## 3. 개발자 비하인드: 같은 8가지 타입, 서로 다른 두 개의 파일

여기서부터가 이번 편의 핵심입니다. 커스텀 필드 타입을 정의하고 미리보기를 그리는 코드는 사실 **두 곳에 따로 존재합니다.**

* `src/components/setup/steps/Step7CustomFields.tsx` — 신규 가입 시 셋업 위자드
* `src/pages/AdminPage.tsx` — 가입 이후 관리자 콘솔에서 필드를 추가·수정하는 화면

왜 하나로 합치지 않았느냐고 물을 수 있습니다. 위자드는 "처음 설정하는 흐름"에 맞춰 단계별 애니메이션과 간결한 UI를 쓰고, 관리자 콘솔은 "이미 등록된 필드 목록을 관리"하는 데 최적화된 테이블형 UI를 씁니다. 화면 구조 자체가 다르다 보니 컴포넌트를 억지로 공유하기보다, 타입 정의(`CustomFieldType`)만 공유하고 렌더링은 각자 구현하는 쪽을 택했습니다.

```typescript
// src/types/index.ts
export type CustomFieldType =
  'text' | 'number' | 'select' | 'radio' | 'checkbox' | 'checkbox_group' | 'phone' | 'image_upload'
```

문제는 이 타입에 새 값을 하나 추가할 때마다, 두 파일 안에 있는 `FIELD_TYPE_DEFS`(타입별 라벨·아이콘·배지 정의)와 미리보기 컴포넌트를 **손으로 양쪽 다 고쳐야 한다**는 점입니다. 하나라도 빠뜨리면 위자드에서 만든 필드가 관리자 콘솔에서는 아이콘이 깨져 보이는 식의 불일치가 생깁니다.

> **📸 [이미지 삽입 구간: 위자드와 관리자콘솔의 커스텀 필드 타입 선택 UI 나란히 비교]**
> *설명: 왼쪽은 셋업 위자드 7단계의 필드 타입 아이콘 목록, 오른쪽은 관리자콘솔 커스텀필드 탭의 동일한 목록. 두 화면의 아이콘·라벨이 동일하게 맞춰져 있어야 한다는 걸 보여주는 비교 스크린샷.*

## 4. Troubleshooting: 새 타입을 추가할 때 가장 빠뜨리기 쉬운 배열

### 🔴 위험했던 지점

`image_upload` 타입을 `CustomFieldType`에 추가하던 시점을 되짚어보면, 가장 실수하기 쉬운 함정은 "기존 타입 설정을 복사해서 새 타입에 대충 얹는 것"이었습니다. 드롭다운·라디오·체크박스그룹처럼 "선택지가 있는 타입" 목록(`FIELD_TYPES_WITH_OPTIONS`)과 "대시보드에 집계 가능한 타입" 목록(`FIELD_TYPES_WITH_DASHBOARD`)에 무심코 이미지 타입까지 포함시켰다면 어땠을까요.

필드 편집 화면에는 이미지 타입인데도 "선택지 추가" 입력창이 떴을 것이고, 더 심각하게는 관리자 대시보드가 이미지 파일 URL 문자열을 숫자·비율 통계로 집계하려다 위젯이 깨졌을 겁니다. 이미지 업로드는 애초에 "선택지"라는 개념도 없고, 파일 URL을 통계로 환산할 수도 없는 타입이니까요.

### 🟢 방지책

그래서 이 함정을 코드 리뷰나 기억력에 맡기지 않고, `CLAUDE.md`에 명시적인 규칙으로 못박아뒀습니다.

> **image_upload 등 파일 관련 필드**는 `show_in_dashboard`, `FIELD_TYPES_WITH_OPTIONS`, `PLACEHOLDER_TYPES` 배열에서 제외되어야 하므로, 양쪽 파일 모두에서 이 배열에 추가하지 않는다.

새 타입을 추가할 때 "이 타입이 어떤 배열에 포함돼야 하는가"를 배열 하나하나 점검하지 않으면 이런 실수가 생기기 쉽습니다.

```typescript
// src/types/index.ts
export const FIELD_TYPES_WITH_OPTIONS: CustomFieldType[] = ['select', 'radio', 'checkbox_group']
export const FIELD_TYPES_WITH_DASHBOARD: CustomFieldType[] = ['select', 'radio', 'checkbox_group', 'checkbox']
```

여기에 더해, 플레이스홀더가 의미 없는 타입(드롭다운·라디오·체크박스류)도 별도 배열로 분리해뒀습니다.

```typescript
// Step7CustomFields.tsx
const PLACEHOLDER_TYPES: CustomFieldType[] = ['text', 'number', 'select', 'phone']
```

그래서 커스텀 필드 타입을 하나 추가할 때는 체크리스트를 따라갑니다. "이 타입은 선택지가 필요한가? 대시보드에 집계 가능한가? 플레이스홀더가 의미 있는가? 위자드와 관리자콘솔 양쪽의 정의·아이콘·미리보기를 모두 수정했는가?" 네 가지를 매번 확인합니다. 두 파일이 갈라져 있는 구조 자체를 바꾸기보다, 갈라져 있다는 사실을 놓치지 않는 습관을 만드는 쪽을 택한 셈입니다.
