# [제 3편] 한 칸 안에 두 사람 — 역할(Role) 기반 배정과 셀 분할

봉사 단체를 운영하다 보면 이런 요청을 자주 받습니다. "팀장은 항상 한 명 있어야 하고, 봉사자는 여러 명 배정돼도 되는데, 이 둘을 화면에서 구분해서 보여주면 좋겠어요." 단순히 이름만 나열하는 캘린더로는 부족한 순간입니다.

3편에서는 조직마다 다른 "역할" 개념을 어떻게 데이터로 표현하고, 화면 한 칸 안에서 어떻게 나눠 보여주는지 이야기해보겠습니다.

---

## 1. 실사용 시나리오: 역할이 필요한 조직, 필요 없는 조직

### 시나리오: 팀장 1명 + 봉사자 다수, 유기견 보호소

한 슬롯(예: 토요일 10~12시)에 팀장 1명과 봉사자 3~4명이 함께 배정됩니다. 팀장은 그날 현장을 총괄하는 사람이라 누구인지 한눈에 보여야 하지만, 화면을 팀장 칸/봉사자 칸으로 딱 나눠버리면 봉사자 칸이 너무 좁아집니다.

이럴 때 쓰는 게 **인디케이터 바(indicator_bar)** 역할입니다. 셀을 반으로 쪼개지 않고, 상단에 얇은 띠 형태로 "팀장: 김민수"만 얹어 보여줍니다. 나머지 공간은 그대로 봉사자들 차지입니다.

### 시나리오: 커트팀 / 펌·염색팀, 미용실

반면 미용실은 다릅니다. 같은 시간대에 커트 담당 디자이너와 펌·염색 담당 디자이너가 각각 독립적으로 예약을 받아야 하고, 둘을 섞어서 보여주면 오히려 헷갈립니다. 이럴 땐 **셀 분할(split_cell)** 역할을 씁니다. 한 슬롯이 세로로 반씩 나뉘어 "커트팀"과 "펌·염색팀"이 나란히 보입니다.

같은 "역할"이라는 개념인데 화면에 나타나는 방식은 완전히 다릅니다. 그래서 역할 하나마다 이 둘을 스위치로 켜고 끌 수 있게 만들었습니다.

> **📸 [이미지 삽입 구간: 인디케이터 바 vs 셀 분할 비교]**
> *설명: 왼쪽은 보호소 캘린더 셀(상단에 "팀장: 김민수" 얇은 띠 + 아래에 봉사자 명단), 오른쪽은 미용실 캘린더 셀(세로로 반씩 나뉜 "커트팀"/"펌·염색팀" 칸)을 나란히 배치한 스크린샷.*

### 시나리오: 역할이 아예 필요 없는 조직

물론 대부분의 조직은 역할 구분이 필요 없습니다. 스터디카페 좌석 예약에 "역할"이라는 개념이 끼어들 이유가 없죠. 그래서 역할은 기본적으로 0개이고, 필요한 조직만 추가하는 선택적 기능입니다. 역할이 하나도 없으면 캘린더는 그냥 이름만 나열되는 단순한 형태로 돌아갑니다.

## 2. 관리자 조작 가이드: 역할 만들고 배정 비율 정하기

> **📸 [이미지 삽입 구간: 관리자 콘솔 - 역할 탭, 역할 목록과 추가 폼]**
> *설명: "팀장(인디케이터)", "커트팀(분할)", "펌·염색팀(분할)" 세 역할이 순서대로 나열되고, 각 항목 옆에 위/아래 화살표로 순서 변경 버튼이 있는 화면.*

1. 관리자 콘솔 → **역할** 탭으로 이동합니다.
2. **+ 역할 추가**를 눌러 이름을 입력합니다.
3. 표시 방식을 고릅니다.
   - **칸 분할**: 이 역할만의 독립된 칸이 필요할 때 (예: 커트팀/펌염색팀)
   - **상단 표시줄**: 칸을 나누지 않고 얇은 띠로만 보여주고 싶을 때 (예: 팀장)
4. **고객 정보 필요** 옵션을 켜면, 이 역할로 배정할 때 이름 대신 (또는 이름과 함께) 고객명·연락처를 입력하는 필드가 추가로 뜹니다. 부동산 중개사나 미용실 디자이너처럼 "이 사람이 어떤 고객을 응대하는지"가 중요한 업종에 씁니다.
5. 순서를 화살표로 조정하면 캘린더에 나열되는 순서도 그대로 바뀝니다.

역할을 자동배정에도 반영하고 싶다면, **자동배정** 탭에서 역할별 비율을 정합니다.

> **📸 [이미지 삽입 구간: 관리자 콘솔 - 자동배정 탭, 역할별 비율 슬라이더]**
> *설명: "팀장 20% / 봉사자 80%" 처럼 역할별 배정 비율을 슬라이더로 조정하는 화면. 합계가 100%가 아니면 경고 문구가 뜸.*

6. 각 역할 옆 슬라이더로 비율을 조정합니다. 합이 100%가 되어야 저장이 가능합니다.
7. 자동배정을 실행하면, 빈 슬롯을 채울 때 이 비율에 최대한 맞춰 인원을 분배합니다.

## 3. 개발자 비하인드: 같은 "역할"인데 렌더링이 갈리는 지점

`tenant_roles` 테이블 자체는 단순합니다. `name`, `split_cell`, `indicator_bar`, `requires_customer_info`, `display_order` 다섯 개 컬럼이 전부입니다. 흥미로운 건 이 값을 소비하는 쪽입니다.

```typescript
// src/pages/SchedulePage.tsx
const splitRoles = tenantRoles.filter(r => r.split_cell && !r.indicator_bar)
const indicatorBarRoles = tenantRoles.filter(r => r.indicator_bar)
```

`split_cell`과 `indicator_bar`는 이론적으로는 둘 다 켤 수 있는 별개의 boolean이지만, 실제로는 상호 배타적으로 다룹니다. `indicator_bar`가 켜져 있으면 `split_cell` 값과 무관하게 인디케이터로 취급하는 것이죠. 관리자가 실수로 둘 다 켜더라도 화면이 깨지지 않도록, "우선순위가 있는 두 개의 boolean"을 필터 단계에서 명확히 정리해둔 셈입니다.

캘린더 그리드(`ScheduleGrid`, `WeekGrid`)에는 `splitRoles`만 전달해 칸을 실제로 나누고, `indicatorBarRoles`는 별도 props로 넘겨 셀 상단에 얇은 바로 그립니다. 반면 배정 등록 모달을 열 때는 두 목록을 합쳐서 넘깁니다.

```typescript
// 역할 배정 모달용: split_cell 또는 indicator_bar가 true인 역할 모두 포함
splitRoles={[...splitRoles, ...indicatorBarRoles]}
```

화면에는 다르게 그려지지만, "이 사람이 어떤 역할로 배정되는가"라는 데이터 관점에서는 동등하게 다뤄야 하기 때문입니다. 렌더링 로직과 데이터 로직을 섞지 않고 필요한 지점에서만 목록을 조합하는 방식으로, 두 역할 타입을 하나의 시스템 안에 자연스럽게 공존시켰습니다.

## 4. Troubleshooting: "이름만 바꾸고 싶었는데" — 삭제와 수정의 경계

### 🔴 위험한 지점

관리자가 "커트팀"이라는 역할명을 "헤어팀"으로 바꾸고 싶을 때, 만약 화면에 "이름 수정" 버튼이 따로 없고 "삭제 후 새로 만들기"만 가능하다면 어떻게 될까요. `assignments` 테이블의 각 배정 레코드는 `role_id`로 역할을 참조하고 있어서, 역할을 삭제하는 순간 지난달 배정 기록에서도 역할 이름이 통째로 "이름 없음" 취급을 받게 됩니다. 겉보기엔 단순한 "이름 바꾸기"였는데, 실제로는 과거 데이터까지 건드리는 작업이 되어버리는 셈입니다.

### 🟢 설계로 미리 나눠둔 동선

`role_id`는 `ON DELETE SET NULL`로 걸려 있어 역할이 삭제돼도 배정 레코드 자체는 사라지지 않습니다. 다만 화면에서 역할명을 role_id로 조인해서 보여주다 보니, role_id가 NULL이 되는 순간 "역할 미지정"으로 보이는 건 당연한 결과입니다.

"역할 이름 바꾸기"와 "역할 삭제 후 재생성"은 결과가 전혀 다른 작업입니다. 전자는 기존 role_id를 그대로 두고 `name` 컬럼만 업데이트하면 과거 기록까지 자연스럽게 새 이름으로 보이지만, 후자는 참조가 끊어지며 과거 기록의 역할 표시가 사라집니다. 그래서 역할 관리 화면에는 삭제 버튼과 별개로 **이름만 바꾸는 인라인 수정** 버튼을 둡니다. 관리자가 "이름만 고치고 싶은 것"과 "완전히 지우고 싶은 것"을 화면에서부터 명확히 구분해서 고를 수 있도록 동선을 나눠둔 것입니다.

> **📸 [이미지 삽입 구간: 역할 목록의 이름 수정 vs 삭제 버튼]**
> *설명: 역할 관리 화면에서 "커트팀" 항목 옆에 연필(이름 수정) 아이콘과 휴지통(삭제) 아이콘이 나란히 배치되고, 삭제 아이콘에는 "과거 배정 기록의 역할 표시가 사라집니다" 경고 툴팁이 뜨는 화면.*

```typescript
// updateRole은 role_id를 그대로 유지한 채 name만 갱신 — 과거 배정 기록 영향 없음
const updateRole = useCallback(async (id: string, fields: Partial<...>) => {
  const { error } = await supabase.from('tenant_roles').update(fields).eq('id', id)
  ...
})
```

삭제 버튼에도 "이 역할로 배정된 과거 기록의 역할명이 사라집니다"라는 경고 문구를 추가해, 관리자가 의도치 않게 데이터를 잃는 상황을 사전에 막았습니다.

## 5. 3편을 마치며

같은 boolean 두 개로 완전히 다른 두 가지 화면 표현을 만들어낸 이야기, 그리고 "삭제"와 "이름 변경"을 구분하지 못해 벌어진 실수를 되짚어봤습니다.
