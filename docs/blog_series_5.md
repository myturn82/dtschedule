# [제 5편] 공평하게 나눠주는 법 — 자동배정 알고리즘 설계기

"빈 슬롯을 알아서 채워주세요"라는 요청을 처음 받았을 때, 가장 쉬운 답은 무작위 배정이었습니다. 랜덤으로 아무나 꽂아 넣으면 코드는 3줄로 끝납니다. 그런데 실제로 이렇게 만들었다면, 한 달 뒤 "왜 저는 이번 달에 배정을 5번 받았는데 옆 사람은 1번밖에 안 받았나요?"라는 항의가 들어왔을 겁니다.

5편은 "무작위"가 아니라 "공평함"을 목표로 삼은 자동배정 알고리즘 이야기입니다.

---

## 1. 실사용 시나리오: 배정 횟수가 아니라 "가능 여부"까지 고려하기

### 시나리오: 주중엔 회사 다니는 봉사자들

유기견 보호소 봉사단을 다시 예로 들어보겠습니다. 봉사자 10명 중 절반은 평일 낮에 일을 하고, 나머지 절반은 프리랜서라 평일도 가능합니다. 이 상태에서 단순히 "배정 횟수가 적은 사람부터" 채워 넣으면, 직장인 봉사자가 평일 슬롯에 배정되는 사고가 납니다.

그래서 자동배정은 두 단계로 후보를 좁힙니다.

1. **이 요일에 가능한 사람인가** (개인별 가능 요일 설정)
2. **이번 달 배정 상한을 넘지 않았는가** (개인별 월 배정 횟수 제한)

이 두 조건을 통과한 후보 중에서만 "누적 배정 수가 가장 적은 사람"을 고릅니다. 조건을 만족하는 사람이 아무도 없으면, 억지로 채우지 않고 그 슬롯은 빈 채로 남깁니다. 관리자가 나중에 수동으로 처리하도록 넘기는 것이 무리하게 부적합한 사람을 꽂아 넣는 것보다 낫다고 판단했습니다.

### 시나리오: 역할 비율까지 맞춰야 하는 조직

역할이 있는 조직(3편 참고)은 여기에 조건이 하나 더 붙습니다. "팀장 20%, 봉사자 80%" 같은 비율이 설정돼 있다면, 자동배정은 역할별로 별도의 라운드로빈을 돌립니다. 팀장 후보군 안에서 공평하게, 봉사자 후보군 안에서 공평하게 — 이렇게 역할 단위로 독립적인 공정성을 보장합니다.

> **📸 [이미지 삽입 구간: 회원별 가능 요일·월 배정 횟수 설정 화면]**
> *설명: 관리자 콘솔에서 봉사자 한 명을 선택해 "가능 요일: 토·일" "월 배정 제한: 4회"를 입력하는 화면.*

## 2. 관리자 조작 가이드: 미리보기 없이는 확정하지 않는다

자동배정에서 가장 중요하게 설계한 원칙은 **"실행 즉시 반영하지 않는다"**입니다.

> **📸 [이미지 삽입 구간: 자동배정 미리보기 모달]**
> *설명: 제안된 배정 목록이 "홍길동 - 3/15(토) 10-12시" 형태로 나열되고, 항목별로 체크박스를 해제해 제외할 수 있는 화면. 하단에 "N건 적용" 버튼.*

1. 관리자 콘솔 → **자동배정** 탭에서 배정할 기간(주 단위/월 단위)을 선택합니다.
2. **미리보기 생성**을 누르면, 실제 DB에 아무것도 쓰지 않은 채 제안 목록만 계산해서 보여줍니다.
3. 마음에 들지 않는 제안은 체크박스를 해제해 제외할 수 있습니다.
4. **적용**을 눌러야 비로소 `assignments` 테이블에 실제로 기록됩니다.

이 미리보기 단계가 없으면, 알고리즘이 아무리 정교해도 관리자 입장에선 "블랙박스가 마음대로 사람을 꽂아 넣는" 불안한 기능이 됩니다. 계산과 확정을 분리한 것이 기능 자체의 알고리즘 정교함보다 신뢰를 얻는 데 더 크게 기여했습니다.

## 3. 개발자 비하인드: 라운드로빈, 그리고 "누적 카운트"의 기준점

공정성의 핵심은 결국 "누가 이미 몇 번 배정됐는가"를 정확히 세는 것입니다. 이 카운트는 이번에 새로 제안하는 배정뿐 아니라, **이미 DB에 저장된 기존 배정**까지 포함해야 합니다. 그렇지 않으면 이번 달 초에 이미 3번 배정받은 사람이, 자동배정 실행 시점에는 "0번 배정"으로 취급돼 또 우선적으로 채워지는 역설이 생깁니다.

```typescript
// src/utils/autoAssign.ts
const countMap = new Map<string, number>()
members.forEach(m => countMap.set(m.id, 0))
existingAssignments.forEach(a => {
  if (a.user_id && countMatchFn(a) && countMap.has(a.user_id)) {
    countMap.set(a.user_id, (countMap.get(a.user_id) ?? 0) + 1)
  }
})
```

`countMatchFn`은 호출부에서 주입하는 함수로, "이 배정이 카운트에 포함되는지"를 판단합니다. 예를 들어 역할별로 따로 공정성을 계산해야 하면 `a.role_id === targetRoleId`처럼 좁혀서 넘깁니다. 카운트 로직 자체는 그대로 두고, "무엇을 셀 것인가"만 바꿔 끼우는 구조라 역할 단위 배정과 전체 배정을 같은 함수로 처리할 수 있었습니다.

슬롯을 순회하며 후보를 고르는 부분은 의도적으로 단순하게 짰습니다.

```typescript
let chosen = candidates[0]
let chosenCount = countMap.get(chosen.id) ?? 0
for (const c of candidates) {
  const cnt = countMap.get(c.id) ?? 0
  if (cnt < chosenCount) { chosen = c; chosenCount = cnt }
}
```

동점일 때는 `candidates` 배열의 순서상 먼저 나온 사람이 이깁니다. 정렬 기준을 따로 두지 않고 "안정적인 첫 번째 승자" 방식을 택한 이유는, 매번 실행할 때마다 결과가 미세하게 흔들리는 것보다 예측 가능한 편이 관리자 입장에서 이해하기 쉽다고 판단했기 때문입니다.

## 4. Troubleshooting: 미리보기와 실제 저장이 어긋날 뻔한 지점

### 🔴 처음에 걱정했던 시나리오

이 기능을 설계하면서 가장 경계했던 실수는 "미리보기 = 계산 결과"와 "적용 = 그 계산 결과를 그대로 저장"을 서로 다른 두 번의 계산으로 만들어버리는 것이었습니다. 미리보기를 띄울 때 한 번 계산하고, "적용" 버튼을 눌렀을 때 처음부터 다시 계산한다면 어떻게 될까요? 그 사이에 다른 관리자가 같은 슬롯 몇 개에 수동으로 배정을 넣어버리면, 적용 시점에 재계산된 결과가 사용자가 미리보기에서 확인하고 승인한 내용과 달라질 수 있습니다. "체크박스로 골라낸 항목"과 "실제로 저장되는 항목"이 어긋나는, 발견하기도 어려운 종류의 버그입니다.

### 🟢 설계로 미리 막은 방식

그래서 애초에 계산은 미리보기 단계에서 딱 한 번만 하도록 만들었습니다. `SchedulePage.tsx`에서 `computeAutoAssignments()`를 호출해 나온 `ProposedAssignment[]` 배열을 컴포넌트 상태(`autoProposals`)에 그대로 담아 모달에 props로 넘기고, 모달은 이 배열을 재계산하지 않습니다.

```typescript
// src/components/modals/AutoAssignPreviewModal.tsx
const selected = proposals.filter(p => !excluded.has(p.id))
```

사용자가 체크박스를 해제해 제외한 항목의 id만 `excluded` 셋에 쌓이고, "적용"을 누르면 이 필터링된 배열을 그대로 insert 쿼리에 사용합니다. 재계산이 아예 일어나지 않으니, 사용자가 화면에서 본 것과 실제로 저장되는 것이 어긋날 여지 자체가 없습니다.

물론 이 방식에도 한 가지 빈틈은 남습니다. 미리보기를 띄워둔 사이 다른 관리자가 같은 슬롯에 수동 배정을 넣으면, 적용 시점에 중복 배정 시도가 생길 수 있습니다. 이건 계산 로직으로 막을 수 없는 영역이라, DB의 고유 제약조건(같은 조직·날짜·시간대·이름 조합은 유니크)을 최종 방어선으로 남겨뒀습니다. 중복 시도는 그 건만 에러로 걸러지고 나머지 정상 건은 그대로 저장됩니다. "계산 일관성은 단일 계산으로, 동시성 충돌은 DB 제약조건으로"라는 두 겹의 방어로 정리한 셈입니다.

## 5. 5편을 마치며

공평함이라는 다소 추상적인 목표를, "누적 카운트가 가장 적은 후보 선택"이라는 구체적인 규칙으로 옮긴 이야기와 미리보기·적용 사이의 일관성 문제를 다뤘습니다.

다음 **[제 6편]**에서는 배정 등록 시 사진 한 장을 첨부하는 기능이 왜 필요했는지, 그리고 원본 이미지를 그대로 저장하지 않고 브라우저에서 직접 압축하도록 설계한 이유를 살펴보겠습니다.
