# 변경사항 점검 체크리스트 템플릿

기능 추가/수정 작업이 끝나면 `docs/checklist_YYYY-MM-DD.md` 파일을
이 템플릿을 기준으로 작성하고, 항목별로 직접 동작을 확인하며 체크한다.

기존 예시:
- [docs/checklist_2026-06-11.md](./checklist_2026-06-11.md)
- [docs/test-customer-deletion.md](./test-customer-deletion.md) (기능별 상세 체크리스트)

---

## 작성 규칙

1. 파일명: `docs/checklist_YYYY-MM-DD.md` (날짜는 작업 완료일 기준)
2. 테스트 환경(URL, 계정 등)을 최상단에 명시한다.
3. 변경된 기능/화면 단위로 섹션을 나눈다.
4. 각 항목은 `- [ ]` 체크박스로 작성하고, 사용자가 직접 클릭하며 동작을 확인한다.
5. UI 동작뿐 아니라 DB 반영 결과도 확인이 필요하면 `**[DB 확인]**` 표시를 붙인다.
6. 운영(prod) 반영이 포함된 경우 `**[운영 DB]**` 항목을 별도로 둔다.
7. 기존 기능에 영향을 줄 수 있는 변경이면 "회귀 테스트" 섹션을 추가한다.
8. 마지막에 "테스트 우선순위" 섹션으로 점검 순서를 정리한다.

---

## 템플릿

```markdown
# YYYY-MM-DD 변경사항 점검 체크리스트

테스트 환경: http://localhost:5173 (또는 운영 URL)
테스트 계정: <필요 시 명시>

---

## 1. <변경/기능 이름>

배경: <왜 이 변경이 필요했는지 1~2줄>

- [ ] <정상 시나리오 1>
- [ ] <정상 시나리오 2>
- [ ] <예외/에러 케이스>
- [ ] **[DB 확인]** <DB에 반영되어야 하는 값>

---

## 2. <다른 변경/기능 이름>

- [ ] ...

---

## N. 회귀 테스트 (기존 기능 영향 확인)

- [ ] <이번 변경으로 영향받을 수 있는 기존 기능 1>
- [ ] <이번 변경으로 영향받을 수 있는 기존 기능 2>

---

## N+1. DB / 마이그레이션 반영 상태 (스키마 변경이 있는 경우)

- [ ] 개발 DB(`mcuszdvophmqrwostcah`) 마이그레이션 적용 확인
- [ ] **[운영 DB]** `npx supabase migration list --linked` 결과 Local/Remote 일치 (운영 반영 후)
- [ ] `supabase/reset_db.sql` / `reset_data.sql` 최신 마이그레이션 번호로 갱신됨
- [ ] `npx tsc --noEmit` 통과

---

## 테스트 우선순위

1. **N번** <가장 핵심적인 변경 — 이유>
2. **N번** <다음 우선순위 — 이유>
```
