# [제 1편] 스프레드시트의 해방 — 기획 및 멀티테넌트 아키텍처 설계

안녕하세요! 기술 블로그 전문 에디터이자 시니어 개발자입니다.

수십 명에서 수백 명이 참여하는 조직의 스케줄을 수작업으로 짤 때 머리가 깨질 듯한 통증을 느껴본 적이 있으신가요? 엑셀의 복잡한 수식과 단톡방 투표의 늪에서 벗어나, 조직의 규칙에 맞게 일정이 알아서 흘러가도록 만드는 **다이나믹팀스케줄(DTS)**의 첫 번째 개발기이자 사용 설명서입니다.

오늘은 첫 순서로, **"왜 이 서비스를 만들었는지(3가지 운영 모드의 실사용 예시)"**, **"어떻게 조직에 맞춰 설정하는지(조작 가이드)"**, 그리고 이를 구현하기 위한 **"멀티테넌트 격리 DB 구조(개발자 비하인드)"**를 살펴보겠습니다.

---

## 1. 실사용 시나리오: 우리 조직은 어떤 모드를 써야 할까요?

DTS의 가장 강력한 특징은 단체의 성격에 맞춰 **세 가지 운영 모드** 중 하나를 선택할 수 있다는 점입니다. 실제 도입 사례를 통해 각 모드가 어떤 불편을 해결하는지 알아봅니다.

### 💡 시나리오 A: 회원공유 모드 (Shared Mode)
* **어떤 조직인가요?** 주말 유기견 보호소 봉사단
* **기존의 불편함**: 카페 공지사항 댓글로 "누구누구 토요일 오전 갈게요"라고 적다 보니, 특정 시간대에 20명이 몰리고 정작 일요일 오후에는 아무도 안 와서 봉사 관리가 안 되었습니다.
* **DTS 해결책**: **회원공유 모드**를 사용하면, 모든 회원이 캘린더를 보고 "누가 어디에 배정되어 있는지"를 파악할 수 있습니다. 정원이 남은 빈 슬롯을 직접 클릭하여 본인의 참여를 등록하고, 혹시 일이 생기면 동료가 대신 갈 수 있도록 일정을 직접 조율합니다.

### 💡 시나리오 B: 회원개별 모드 (Individual Mode)
* **어떤 조직인가요?** PT 헬스장 또는 1:1 심리상담 센터
* **기존의 불편함**: 타인의 예약 현황(이름, 시간 등)이 그대로 노출되면 개인정보 침해 소지가 있고, 회원이 임의로 다른 사람의 예약을 변경하는 불상사가 발생할 수 있었습니다.
* **DTS 해결책**: **회원개별 모드**를 적용하면, 회원은 로그인했을 때 **오직 본인의 배정 내역**만 볼 수 있습니다. 다른 사람의 예약 시간은 "예약 완료" 상태로만 노출되며, 배정 추가/수정은 본인 일정 영역에 한해서만 가능하거나 전용 조회 모드로 제어됩니다.

### 💡 시나리오 C: 비회원 모드 (Freeform Mode)
* **어떤 조직인가요?** 대학가 소규모 카페 알바 스케줄링
* **기존의 불편함**: 알바생들에게 "일일이 회원가입 하라"고 하니 귀찮다며 안 하고, 점장님이 직접 엑셀로 짜서 단톡방에 매주 이미지로 공유해야 했습니다.
* **DTS 해결책**: **비회원 모드**를 활성화합니다. 알바생들은 회원가입을 할 필요가 전혀 없습니다. 점장님(관리자)이 관리자 콘솔에서 알바생들의 이름을 임의로 적어 스케줄을 완성한 뒤, **"공유 링크"**만 단톡방에 던집니다. 비회원들은 로그인 없이 그 링크를 눌러 완성된 캘린더를 바로 확인합니다.

### 💡 시나리오 D: 비회원 모드 — 소규모 미용실 직원 스케줄 + 고객 예약 안내
* **어떤 조직인가요?** 직원 3~4명 규모의 동네 미용실
* **기존의 불편함**: 원장님이 매주 카카오톡으로 "이번 주 A 디자이너 화·목 출근, B 디자이너 수·금 출근" 식으로 공지를 보내지만, 고객이 전화해서 "오늘 A 디자이너 있어요?"라고 물어보는 일이 반복됩니다. 예약 장부는 종이 수첩이어서 실수가 잦고, 직원들도 서로 스케줄을 확인하려면 원장님에게 다시 물어봐야 했습니다.
* **DTS 해결책**: **비회원 모드**로 직원 출근 스케줄과 담당 메뉴(커트·펌·염색)를 슬롯별로 입력합니다. 공유 링크를 인스타그램 프로필이나 네이버 스마트플레이스에 걸어두면, 고객이 로그인 없이 이번 주 담당 디자이너 일정을 직접 보고 전화 예약까지 이어집니다. 직원들은 같은 링크로 본인 스케줄을 언제 어디서든 확인합니다.

### 💡 시나리오 E: 회원개별 모드 — 부동산 중개사무소 매물 방문 예약
* **어떤 조직인가요?** 공인중개사 5명이 함께 일하는 부동산 사무소
* **기존의 불편함**: 중개사마다 고객 상담·매물 방문 약속이 따로 있는데, 사무소 공용 일정 앱을 쓰면 다른 중개사의 고객 정보와 거래 일정이 다 보여 내부 경쟁 문제가 생깁니다. 대표원장만 전체를 파악해야 하는데 마땅한 도구가 없어 화이트보드에 손으로 적는 식이었습니다.
* **DTS 해결책**: **회원개별 모드**를 적용합니다. 각 중개사는 로그인 후 **본인 담당 고객의 방문 약속**만 캘린더에서 확인합니다. 다른 중개사의 일정은 "예약 있음" 블록으로만 표시되어 시간대 충돌 여부만 파악됩니다. 대표원장은 관리자 계정으로 전체 일정과 중개사별 성약 건수를 한눈에 조회합니다.

### 💡 시나리오 F: 회원공유 모드 — 방문 요양 서비스 케어 매니저 배정
* **어떤 조직인가요?** 요양보호사 10명이 활동하는 재가 방문 요양 센터
* **기존의 불편함**: 담당 어르신이 갑자기 응급 상황이 생기거나 요양보호사가 몸이 아파 결근할 때, 대체 인력을 카카오톡 단체방에서 "누구 대신 가줄 수 있어요?"라고 묻는 방식으로 구하다 보니 골든타임이 지나 버립니다. 누가 지금 어느 어르신 댁을 방문 중인지 파악도 안 됩니다.
* **DTS 해결책**: **회원공유 모드**로 시간대별 담당 어르신과 방문 요양보호사를 배정합니다. 공백 슬롯이 생기면 다른 팀원이 캘린더에서 직접 확인하고 자원하여 채울 수 있습니다. 센터장은 월별 집계 뷰로 요양보호사별 방문 시간을 집계해 급여 정산 근거 자료로 활용합니다.

### 💡 시나리오 G: 비회원 모드 — 주민센터·공공기관 민원 상담 예약
* **어떤 조직인가요?** 구청 내 복지 상담 창구 (상담사 3명 운영)
* **기존의 불편함**: 전화로만 예약을 받다 보니 업무 시간 중 전화 응대에 너무 많은 시간이 소요됩니다. 수기 예약 장부는 줄 긋고 수정하는 흔적 때문에 다음 상담 날짜를 착각하는 일이 반복되고, 민원인도 예약 확인을 위해 다시 전화해야 합니다.
* **DTS 해결책**: **비회원 모드**로 담당 상담사별 예약 슬롯(30분 단위)을 열어둡니다. 공유 링크를 구청 홈페이지와 문자 알림에 삽입하면, 민원인은 회원가입 없이 링크 하나로 빈 시간대를 확인합니다. 상담사는 관리자 콘솔에서 직접 민원인 이름과 상담 주제를 입력해 예약을 확정하고, 해당 슬롯이 채워지는 것을 공유 뷰에서 즉시 확인합니다.

### 💡 시나리오 H: 회원공유 모드 — 스터디카페·공유 작업실 좌석 예약
* **어떤 조직인가요?** 월정액 회원제로 운영하는 스터디카페 (좌석 20석)
* **기존의 불편함**: 회원들이 특정 시간대에 특정 좌석을 선호해서 "선점 싸움"이 빈번합니다. 카운터 직원이 엑셀로 좌석 배정을 관리하다 보니 실수로 더블 부킹이 나거나, 인기 좌석이 실제로 예약됐는지 비어있는지 실시간으로 알 방법이 없습니다.
* **DTS 해결책**: **회원공유 모드**를 도입합니다. 슬롯을 좌석 번호(A-01, A-02…) 단위로 설정하고 회원들이 직접 원하는 시간대의 빈 좌석을 클릭해 본인 이름으로 예약합니다. 전체 배정 현황이 실시간으로 공유되므로 더블 부킹이 원천 차단됩니다. 관리자는 특정 좌석을 정기 점검이나 청소 일정으로 잠금 처리하여 해당 시간대의 예약을 막을 수 있습니다.

### 💡 시나리오 I: 회원공유 모드 — 테니스클럽 코트 예약
* **어떤 조직인가요?** 코트 2면을 공유하는 동호회 (회원 40명)
* **기존의 불편함**: 회원들이 단톡방에 "토요일 저녁 코트 잡을게요"라고 올리면 댓글로 번호표를 매기는 식이었는데, 인기 시간대(저녁 7~9시)는 매번 먼저 쓴 사람이 임자라 다툼이 잦았고, 총무는 매주 누가 몇 번 코트를 썼는지 정산하느라 골머리를 앓았습니다.
* **DTS 해결책**: **회원공유 모드**로 코트별·시간대별 슬롯을 열어둡니다. 회원은 캘린더에서 빈 코트를 직접 클릭해 예약하고, 전체 회원이 실시간으로 예약 현황을 확인하므로 중복 예약 시비가 사라집니다. 총무는 월말에 회원별 코트 사용 횟수를 집계해 정기 회비 정산 자료로 바로 활용합니다.

### 💡 시나리오 J: 비회원 모드 — 도자기 공방 원데이클래스 예약
* **어떤 조직인가요?** 강사 1명이 운영하는 소규모 공방
* **기존의 불편함**: 인스타그램 DM으로 "몇 시 클래스 자리 있나요?"라는 문의가 하루에도 수십 건씩 오는데, 수강생 명단은 강사가 메모장에 손으로 적다 보니 같은 시간대에 실수로 두 명을 받는 사고가 생겼습니다.
* **DTS 해결책**: **비회원 모드**로 회차별 정원을 설정한 슬롯을 만듭니다. 공유 링크를 인스타그램 프로필 링크에 걸어두면, 수강 희망자는 회원가입 없이 빈 자리를 바로 확인하고 강사는 DM으로 받은 예약자만 관리자 콘솔에 이름을 채워 넣습니다. 정원이 차면 슬롯이 자동으로 마감 처리되어 중복 예약이 원천 차단됩니다.

### 💡 시나리오 K: 회원개별 모드 — 동물병원 진료 예약
* **어떤 조직인가요?** 수의사 2명이 진료하는 동네 동물병원
* **기존의 불편함**: 전화 예약 장부에 보호자 이름과 반려동물 진료 사유를 함께 적어두다 보니, 대기실 모니터에 다른 손님의 진료 내용이 노출되는 게 부담스럽다는 항의가 있었습니다. 특정 수의사만 찾는 단골도 많은데 담당의별 예약 현황을 따로 파악할 방법이 없었습니다.
* **DTS 해결책**: **회원개별 모드**를 적용해 담당 수의사별로 슬롯을 나눕니다. 보호자는 로그인 후 원하는 수의사와 시간대를 골라 예약하고, 본인의 예약 내역만 확인할 수 있어 다른 반려동물의 진료 사유가 노출되지 않습니다. 병원은 담당의별 예약 건수를 한눈에 파악해 진료 시간 배분을 조정합니다.

### 💡 시나리오 L: 회원공유 모드 — 교회 소그룹 모임 장소 예약
* **어떤 조직인가요?** 소그룹(구역) 12개가 한 건물을 함께 쓰는 교회
* **기존의 불편함**: 소그룹 리더들이 카카오톡 단체방에 "이번 주 수요일 2층 소모임실 쓸게요"라고 올리는 식이라, 같은 공간을 두 그룹이 동시에 예약해 버리는 일이 반복됐고, 관리 집사님이 매주 누가 어디를 쓰는지 취합하느라 애를 먹었습니다.
* **DTS 해결책**: **회원공유 모드**로 모임 공간(소모임실, 친교실 등)을 슬롯 단위로 열어둡니다. 소그룹 리더는 캘린더에서 빈 공간과 시간을 직접 선택해 모임을 등록하고, 전체 리더가 서로의 예약 현황을 실시간으로 확인하므로 겹침 없이 조율됩니다. 관리 집사님은 월간 뷰 하나로 전체 공간 사용 현황을 파악합니다.

### 💡 시나리오 M: 비회원 모드 — 퀵서비스·택배 기사 배차 스케줄
* **어떤 조직인가요?** 지입 기사 15명을 운영하는 소규모 물류 대행업체
* **기존의 불편함**: 배차 담당자가 매일 아침 전화로 기사님들의 가용 여부를 일일이 확인한 뒤 수기로 배차표를 작성해 단톡방에 사진으로 올렸는데, 기사님이 사진을 놓치거나 오후 배차 변경 사항을 뒤늦게 확인해 배송이 지연되는 일이 잦았습니다.
* **DTS 해결책**: **비회원 모드**로 시간대별·권역별 배차 슬롯을 만듭니다. 배차 담당자가 관리자 콘솔에서 기사 이름을 슬롯에 배정하면, 기사님은 회원가입 없이 공유 링크 하나로 본인의 오늘 배차 현황을 실시간으로 확인합니다. 배차가 변경되면 캘린더가 즉시 갱신되므로 사진 공지를 놓쳐 발생하는 지연이 사라집니다.

## 3. 개발자 비하인드: 멀티테넌트 RLS와 가변 설정 설계

SaaS 웹 서비스를 설계할 때 가장 중요한 도전 과제는 **"A 조직의 회원이 API를 임의로 조작하여 B 조직의 극비 스케줄을 탈취하거나 변조하는 보안 차단"**이었습니다.

DTS는 Supabase(PostgreSQL)의 **Row Level Security (RLS)**와 **JSONB 컬럼**을 활용하여 서버리스 구조임에도 단 한 줄의 격리 누수 없이 이를 깔끔하게 극복했습니다.

### ① RLS를 통한 물리적 차단막 구축
DB 레벨에서 사용자의 토큰 정보와 로우의 테넌트 식별자(`tenant_id`)를 대조하여 권한이 없는 조회를 즉각 차단합니다. 

```sql
-- assignments 테이블 RLS 정책 설정 예시
CREATE POLICY select_tenant_assignments ON assignments
  FOR SELECT
  USING (
    tenant_id = (SELECT current_setting('request.jwt.claims', true)::jsonb->>'tenant_id')
  );
```
이 정책이 켜져 있으면, 프론트엔드에서 아무리 악의적으로 `select * from assignments` 쿼리를 보내더라도, 로그인한 유저 본인의 조직 데이터 외에는 DB가 절대 0건으로 리턴하여 보장합니다.

### ② JSONB 컬럼을 활용한 다이나믹 스케마 격리
모드 설정, 역할 이름, 테마 컬러 등 테넌트마다 천차만별인 설정값을 저장하기 위해 매번 테이블 컬럼을 추가할 수는 없습니다.  
우리는 `tenants` 테이블에 `settings`라는 **JSONB 컬럼**을 선언하고, [TenantContext.tsx](file:///d:/claudePrj/dtschedule/src/contexts/TenantContext.tsx)에서 전역적으로 상태를 구독하도록 구현했습니다.

## 4. Troubleshooting: 비회원 공유 뷰와 RLS의 충돌 및 극복

개발 중 가장 큰 삽질은 **"비회원 모드에서의 공유 링크 기능"**에서 발생했습니다.

### 🔴 문제 상황
`/share` 라우트 자체는 비로그인 사용자도 열 수 있게 열어뒀지만, 데이터베이스 RLS 정책은 오직 **"로그인한 조직 멤버(`is_tenant_member`)"**만 `assignments`·`schedule_rules` 등을 조회하도록 짜여 있었습니다.  
결과적으로, 회원공유·회원개별 모드는 물론 **비회원 모드조차** 공유 링크로 들어온 비로그인 방문자는 캘린더 화면이 하얗게 비어 보이는(혹은 프론트엔드에서 아예 로그인부터 요구하는) RLS 보안 차단 이슈에 직면했습니다. 그렇다고 RLS를 통째로 비활성화하면 회원공유·회원개별 모드의 조직 격리와 프라이버시가 뚫리는 모순적인 상황이었습니다.

### 🟢 해결 과정
핵심 아이디어는 "**비회원(프리폼) 모드 테넌트에 한해서만** anon 조회를 허용하는 예외 정책을 추가한다"는 것이었습니다. 회원공유·회원개별 모드는 프라이버시가 존재 이유이므로 그대로 로그인 필수로 남겨두고, 애초에 회원 개념이 없는 비회원 모드만 익명 접근 경로를 열어주는 방식입니다.

```sql
-- 해당 테넌트가 비회원(프리폼) 모드인지 판별하는 SECURITY DEFINER 함수
CREATE OR REPLACE FUNCTION public.is_freeform_tenant(tid uuid)
RETURNS boolean
LANGUAGE sql SECURITY DEFINER STABLE
SET search_path = public AS $$
  SELECT EXISTS (
    SELECT 1 FROM tenants
    WHERE id = tid
      AND is_active = true
      AND settings->>'tenant_mode' = '비회원'
  );
$$;

-- 기존 "로그인한 조직 멤버" 정책은 그대로 두고, 비회원 모드 전용 조회 경로를 하나 더 추가
CREATE POLICY "assignments_public_share_select" ON assignments
  FOR SELECT USING (is_freeform_tenant(tenant_id));
```

PostgreSQL RLS는 같은 테이블에 permissive 정책이 여러 개 있으면 **OR로 결합**됩니다. 그래서 기존 `is_tenant_member` 기반 정책을 건드릴 필요 없이, "이 요청이 비회원 모드 테넌트를 향한 것이라면 통과"라는 정책 하나만 얹으면 됩니다. 프론트엔드에서도 `SharePage.tsx`가 대상 테넌트의 `tenant_mode`를 먼저 확인해, 비회원 모드일 때만 로그인 게이트를 건너뛰도록 수정했습니다.

이렇게 회원공유·회원개별 모드의 격리는 그대로 유지하면서, 비회원 모드만 정확히 골라 익명 열람을 허용하는 최소 권한 원칙의 해결책을 완성했습니다.
