# 운영모드 추천 — 추가 후보 (구현 전 검토용)

현재 조직위자드 2단계에서 업종별로 추천하는 운영모드는 3가지다 (`src/components/setup/steps/Step2Mode.tsx`, `src/utils/wiz-data` 계열의 `MODE_CARDS`/`TOP_REC` 참고).

| 모드 | 설명 |
|------|------|
| 회원공유 | 회원들이 스케줄을 실시간으로 공유. 관리자와 멤버가 서로의 배정 현황을 함께 확인 |
| 회원개별 | 관리자만 각 회원의 스케줄을 통합 관리. 회원 간에는 서로의 스케줄이 보이지 않음 |
| 비회원 | 회원 등록 없이 방문자 정보를 직접 입력 (예약·방문) |

이 3가지는 모두 **"관리자가 배정하고, 그 결과를 누구에게 보여주는가"** (가시성·소유권) 축으로 짜여 있다. 이 축으로 잡히지 않는 예외 케이스를 정리한다.

## 후보 1 — 회원 자율예약형 (선착순 신청)

관리자가 배정하는 게 아니라, **회원이 직접 빈 슬롯을 클릭해 본인을 등록**하는 방식.

- 해당 업종 예시: 스터디룸·공유오피스 좌석 예약, 코트/헬스장 PT 슬롯 선점
- 기존 "회원공유"는 관리자가 배정 → 회원이 보기만 하는 흐름에 가까워서, 회원이 스스로 신청하는 업종엔 안 맞을 수 있음
- 참고: 한 슬롯에 여러 명이 들어가는 좌석/자원 예약 자체는 이미 `max_capacity`(`src/types/index.ts`)로 커버되고 있음 — 그건 별도 모드가 아니라 기존 모드 위에 얹히는 설정으로 봐도 됨
- 필요한 것: "누가 배정 가능한가"(관리자 전용 vs 회원 자율) 권한 토글, 취소·노쇼 정책

## 후보 2 — 혼합형 (직원 운영 + 고객 예약 동시)

식당·미용실처럼 **"직원 근무표"(회원공유/개별)** 와 **"손님 예약"(비회원)** 이 한 조직 안에 동시에 필요한 경우.

- 4번째 모드 카드를 추가하기보다, "한 조직에 두 모드를 동시 적용 허용"하는 구조 쪽이 더 맞을 가능성
- 영향 범위가 큼 — 모드 선택 UI뿐 아니라 데이터 모델·권한 체계까지 손볼 필요가 있어 보임

## 후보 3 (애매함) — 교대·당직 순환형

병원 당직, 보안 교대처럼 **규칙 기반으로 자동 순환 배정**되는 형태.

- "누가 보는가"의 모드 축이 아니라 "자동배정 규칙" 축에 가까움
- 모드 추가보다는 5단계(운영 규칙) 쪽 기능으로 푸는 게 더 적절해 보임 — 별도 검토 필요

## 우선순위 의견

- **후보 1**이 가장 명확한 빈틈 — 다음에 더 구체화할 때 우선 검토
- **후보 2**는 모드 카드보다 "모드 조합 허용" 설계가 필요해서 더 큰 결정 — 별도 논의 필요
- 후보 3은 모드 축이 아닐 가능성이 높음 — 운영 규칙 기능 쪽에서 다시 검토

구현은 보류, 추가 후보나 우선순위 변경 시 이 문서를 갱신한다.
