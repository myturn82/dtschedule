# [제 10편] "내일 배정 있는 거 깜빡했어요" — D-1 알림과 무료 스케줄러의 배신

아무리 완벽한 캘린더를 만들어도, 사람들은 캘린더를 매일 들여다보지 않습니다. "내일 오전 배정이 있었는데 깜빡하고 안 나왔어요"라는 연락을 몇 번 받고 나서야 절실히 깨달았습니다. 배정을 아무리 잘 짜도, 그 사실을 하루 전에 다시 한번 알려주지 않으면 무용지물이 될 수 있다는 걸요.

10편은 이 D-1(하루 전) 알림 시스템을 만든 이야기이자, 무료로 쓸 수 있는 스케줄러를 믿었다가 뒤통수를 맞은 이야기이기도 합니다.

> ℹ️ **읽기 전에**: 이 글은 알림 시스템을 처음 설계하고 만든 순서 그대로 기록한 글입니다. 처음엔 인앱 알림과 웹푸시 두 채널, 자동 정시 발송으로 시작했지만, 4장의 Troubleshooting에서 다루는 사건 이후 **웹푸시와 자동 발송은 걷어내고 인앱 알림 수동 발송만 남긴 상태**로 운영 중입니다. 왜 그렇게 됐는지가 이 글의 핵심입니다.

---

## 1. 실사용 시나리오: 인앱 알림과 웹푸시, 두 가지 채널로 시작했다

### 시나리오: 방문 요양 센터의 전날 알림

1편에서 다룬 방문 요양 서비스(시나리오 F)를 다시 가져와 보겠습니다. 요양보호사가 내일 몇 시에 어느 어르신 댁을 방문하는지, 전날 저녁에 한 번 더 알려주는 것만으로도 실수를 크게 줄일 수 있습니다.

처음 설계는 두 채널이었습니다. 앱을 열어야 보이는 **인앱 알림**(종 아이콘 눌러 확인)과, 앱을 열지 않아도 휴대폰 알림창에 뜨는 **웹푸시**. 조직 관리자가 배정알림 탭에서 발송 시간(예: 매일 저녁 6시)과 수신 대상(배정된 회원만/관리자 포함)을 설정해두면, 그 시간에 맞춰 내일 배정된 사람들에게 자동으로 두 채널 모두 알림이 나가는 구조였습니다.

> **📸 [이미지 삽입 구간: 휴대폰 잠금화면에 뜬 웹푸시 알림 + 앱 안의 종 아이콘 알림 패널]**
> *설명: 웹푸시가 살아있던 당시 "📅 내일 배정 알림 - 안녕하세요 OO님! 내일 10시 배정이 있습니다" 알림이 휴대폰 잠금화면에 뜬 모습(왼쪽), 같은 내용이 앱 상단 종 아이콘을 눌렀을 때 목록으로 보이는 화면(오른쪽). 현재는 왼쪽 채널이 비활성화됐다는 점을 캡션으로 명시.*

### 관리자가 직접 지금 바로 보낼 수도 있다

자동 발송을 기다리지 않고 "지금 당장 보내고 싶다"는 요구도 있었습니다. 배정알림 탭의 **지금 발송** 버튼을 누르면, 현재 설정 기준으로 내일 배정된 사람들에게 즉시 알림이 나갑니다. 그리고 관리자는 **발송내역**에서 누구에게 언제 어떤 메시지가 갔고, 읽었는지 여부까지 확인할 수 있습니다. 지금은 이 "지금 발송" 버튼이 유일한 발송 경로가 됐습니다.

## 2. 관리자 조작 가이드: 지금은 이렇게 씁니다 (수동 발송 기준)

> **📸 [이미지 삽입 구간: 관리자 콘솔 - 배정알림 탭 전체 화면 (현재 버전)]**
> *설명: "자동 정시 발송은 현재 잠시 중단된 상태입니다" 안내 문구, 수신 대상 체크박스, 메시지 템플릿 입력창, 그 아래 "지금 발송" 버튼과 발송내역 목록이 순서대로 배치된 화면.*

1. 관리자 콘솔 → **배정알림** 탭으로 이동합니다. 상단에 "자동 정시 발송은 현재 잠시 중단된 상태입니다"라는 안내가 보입니다.
2. **수신 대상**에서 배정된 멤버/관리자 중 알림 받을 대상을 선택합니다.
3. **메시지 템플릿**에 `{{name}}`, `{{date}}`, `{{slot}}`, `{{org}}` 같은 변수를 섞어 원하는 문구를 작성해둡니다. (활성화 토글·발송 시간 항목은 자동 발송이 재개될 때를 대비해 미리 준비해두는 용도이고, 지금은 실제 발송에 영향을 주지 않습니다.)
4. 내일 배정자에게 알릴 시점이 되면 **지금 발송** 버튼을 직접 누릅니다. 이 버튼을 누르는 순간에만 실제로 알림이 나갑니다.
5. 하단 **발송내역**에서 수신자·메시지·발송일시·읽음 여부를 확인하고, 이름·읽음 상태·날짜로 필터링할 수 있습니다.

> **📸 [이미지 삽입 구간: 발송내역 목록 + 필터 UI]**
> *설명: 수신자 이름 검색창, "전체/읽음/안읽음" 드롭다운, 날짜 선택 입력이 상단에 나란히 있고, 그 아래 카드형 발송 기록이 세로로 나열된 화면. 필터를 걸었을 때 목록이 줄어드는 모습.*

## 3. 개발자 비하인드: 웹푸시는 왜 서버가 암호화까지 해야 하는가

인앱 알림은 단순합니다. `notifications` 테이블에 행 하나를 insert하면 끝입니다. 문제는 웹푸시입니다. 브라우저가 꺼져 있어도 OS 알림창에 뜨게 하려면, 서버가 **Web Push 프로토콜(RFC 8291)**을 직접 구현해야 합니다. 별도 SaaS(파이어베이스 등)를 쓰지 않고 Supabase Edge Function 안에서 WebCrypto만으로 이 프로토콜을 구현했습니다.

핵심은 VAPID(서버 신원을 증명하는 키페어)로 서명한 JWT를 만들고, 알림 내용을 ECDH 키 교환 + AES-128-GCM으로 암호화해 브라우저의 푸시 서비스(FCM, Mozilla Push 등)로 전달하는 흐름입니다.

```typescript
// supabase/functions/send-reminders/index.ts (요약)
const jwt = `${jwtHeader}.${jwtPayload}.${sig}`  // VAPID 개인키로 서명
const body = await encryptWebPushPayload(payload, p256dhBase64, authBase64)  // RFC 8291 암호화
const res = await fetch(endpoint, {
  method: 'POST',
  headers: { Authorization: `vapid t=${jwt},k=${vapidPublic}`, 'Content-Encoding': 'aes128gcm' },
  body,
})
```

배치 자체는 발송 대상을 찾는 로직이 단순합니다. `notification_settings.send_time`이 현재 서울 시각과 정확히 일치하는 조직만 골라, 내일 날짜(`getTomorrowSeoul()`)로 배정된 사람을 찾아 인앱 insert + 웹푸시 전송을 순서대로 실행합니다. 이 배치를 매일 정해진 시간에 깨워주는 역할은 **GitHub Actions**가 맡았습니다. 무료로, cron 문법 한 줄로 예약 실행을 걸 수 있는 손쉬운 선택지였습니다.

## 4. Troubleshooting: 무료 스케줄러는 "무료"인 이유가 있었다

### 🔴 문제 상황

GitHub Actions의 `schedule` 트리거로 매시 정각(나중엔 30분 간격)에 Edge Function을 호출하도록 워크플로우를 짜뒀습니다. 그런데 실제 실행 이력을 확인해보니, 최근 실행이 전부 `failure`로 끝나고 있었습니다. 원인은 워크플로우에 필요한 저장소 시크릿(`SUPABASE_URL`) 값이 아예 비어있었던 것이었습니다. curl 명령이 빈 문자열과 경로를 이어붙여 만든 `"/functions/v1/send-reminders"`라는, 호스트가 빠진 기형적인 URL을 호출하려다 매번 `curl: (3) URL malformed` 에러로 종료되고 있었습니다.

시크릿을 채워 넣어 이 문제는 해결했습니다. 그런데 워크플로우를 수동으로 실행(`workflow_dispatch`)해 정상 동작을 확인한 뒤, 진짜 문제를 발견했습니다. **자동 스케줄(`schedule`) 트리거가 5시간 넘게 단 한 번도 실행되지 않고 있었습니다.** 워크플로우 상태는 분명 `active`였고, cron 문법도 틀리지 않았는데도요.

과거 실행 이력을 시간순으로 늘어놓아보니 더 뚜렷해졌습니다. 매시 정각으로 설정했을 때조차, 실제 실행 간격은 03:45, 06:36, 09:04, 11:13, 13:27, 14:59, 16:06, 18:00, 20:06... 식으로 1~3시간씩 들쭉날쭉했습니다. GitHub은 공식적으로 "scheduled workflow는 트래픽이 많을 때 지연되거나 심하면 건너뛸 수 있다"고 문서에 명시해두고 있습니다. 특히 짧은 주기(30분 간격 같은)일수록 이 현상이 두드러집니다.

> **📸 [이미지 삽입 구간: GitHub Actions 실행 이력 목록]**
> *설명: "Send D-1 Reminders" 워크플로우의 Actions 탭 실행 목록. 매시 정각 cron인데도 실행 시각이 03:45, 06:36, 09:04처럼 들쭉날쭉하고, 다수의 항목이 빨간 X(실패) 표시로 나열된 화면.*

### 🟢 결정: 자동화를 잠시 내려놓기

여기서 두 가지 선택지가 있었습니다. Supabase 자체의 `pg_cron` 기반 스케줄링으로 트리거를 옮기거나, 아니면 일단 자동 발송을 끄고 안정적인 대안을 다시 설계할 시간을 버는 것. "정시에 정확히 발송된다"는 신뢰가 깨진 알림 기능은 오히려 없느니만 못하다고 판단해, 우선 후자를 택했습니다.

```yaml
# .github/workflows/send-reminders.yml
on:
  # schedule 트리거 비활성화 — GitHub Actions cron 신뢰성 문제로 자동발송 중단
  # schedule:
  #   - cron: '*/30 * * * *'
  workflow_dispatch:   # 수동 실행만 남김
```

동시에 서버 쪽에서도 웹푸시 전송 자체를 강제로 꺼서, 관리자가 누르는 **지금 발송**은 인앱 알림만 확실하게 나가도록 범위를 좁혔습니다.

```typescript
// supabase/functions/send-reminders/index.ts
// 웹푸시 배치 발송 임시 비활성화 — 자동 cron 자체를 껐고 인앱 알림 수동발송만 유지
const webPushEnabled = false
```

"자동으로, 정확한 시간에" 보내는 것보다 "관리자가 누를 때 확실하게" 보내는 쪽이 지금 시점에는 더 신뢰할 수 있는 기능이라고 판단했습니다. 화려한 자동화보다, 훨씬 단순하지만 100% 예측 가능한 수동 버튼 하나가 나을 때도 있다는 걸 배운 경험이었습니다. 이 사건은 무료 인프라를 선택할 때 "무료"라는 이유만으로 가용성까지 무료 수준을 감내할 수 있는지 따져봤어야 했다는 반성으로 남았습니다.

## 5. 10편을 마치며

암호화 프로토콜을 직접 구현하는 건 예상했던 어려움이었지만, 정작 발목을 잡은 건 "이 배치를 언제 깨울 것인가"라는 훨씬 기초적인 질문이었습니다. 화려한 기능 하나보다, 그 기능이 실제로 약속한 시간에 동작하는지를 검증하는 일이 얼마나 중요한지 새삼 확인한 편이었습니다.

다음 **[제 11편]**에서는 이 시리즈의 마지막 편으로, 다크모드 전환 하나 때문에 겪었던 예상 밖의 브라우저 렌더링 문제와, 서비스 전반의 디자인 원칙을 정리해보겠습니다.
