# [제 9편] 로그인 버튼 하나가 감춘 복잡함 — 소셜 인증과 승인 대기 흐름

회원가입 화면에서 이탈률이 가장 높은 구간이 어디인지 아시나요? 이메일 입력, 비밀번호 확인, 약관 동의를 다 거치는 그 몇 단계입니다. 특히 매주 바뀌는 봉사자나 아르바이트생을 받아야 하는 조직이라면, 이 진입 장벽 하나 때문에 "그냥 카톡으로 하자"는 얘기가 나옵니다.

9편은 이 장벽을 낮추기 위해 붙인 카카오 소셜 로그인, 그리고 그 뒤에 숨어있던 생각보다 복잡한 흐름 제어 이야기입니다.

---

## 1. 실사용 시나리오: "카톡으로 3초 만에" 가입시키기

### 시나리오: 매 학기 바뀌는 대학 동아리 스터디그룹

앞서 다뤘던 스터디카페(1편, 시나리오 H)와 비슷하지만 조금 다른 결의 문제입니다. 학기마다 새로 들어오는 신입 부원들에게 "이메일로 회원가입하고 비밀번호 만들고 이메일 인증까지 해주세요"라고 하면 절반은 그 자리에서 포기합니다. 반면 "카카오 버튼 한 번 누르면 끝"이라면 진입 장벽이 확 낮아집니다.

문제는 신입 부원이 카카오로 가입할 때 **"어느 조직에 들어갈지"**를 함께 알려줘야 한다는 점입니다. 이메일 가입 흐름에서는 이 선택이 자연스럽게 한 화면 안에서 이뤄지는데, 카카오 로그인은 중간에 카카오 사이트로 완전히 이동했다가 돌아오는 방식이라 이 선택을 어디에 어떻게 들고 있을지가 까다로운 문제였습니다.

> **📸 [이미지 삽입 구간: 회원가입 화면 - 이메일 폼 vs 카카오 버튼]**
> *설명: 왼쪽은 이름·이메일·비밀번호·약관 동의를 채우는 이메일 가입 폼, 오른쪽은 노란 카카오 버튼 하나만 있는 화면. 두 흐름의 입력 단계 수 차이를 한눈에 보여주는 비교 스크린샷.*

### 시나리오: 관리자 승인이 꼭 필요한 조직

가입은 쉬워졌지만, 아무나 바로 조직에 들어와서 다른 회원의 스케줄을 보게 둘 수는 없습니다. 그래서 회원가입(이메일이든 카카오든)은 즉시 정회원이 되는 게 아니라 **승인 대기** 상태로 시작합니다. 관리자가 확인하고 승인 버튼을 눌러야 비로소 캘린더에 접근할 수 있습니다.

---

## 2. 관리자 조작 가이드: 승인 대기 목록 확인하고 처리하기

> **📸 [이미지 삽입 구간: 신규 가입자가 보는 승인 대기 화면(PendingPage)]**
> *설명: "OO 조직에 가입 신청이 완료되었습니다. 관리자 승인을 기다리는 중입니다" 문구와 함께, 다른 조직에 추가로 가입 신청할 수 있는 버튼이 있는 화면.*

1. 신규 회원이 가입 신청을 하면, 로그인은 되지만 캘린더 화면 대신 **승인 대기 화면**으로 이동합니다.
2. 관리자는 콘솔 → **승인대기** 탭에서 신청자 목록을 확인합니다.

> **📸 [이미지 삽입 구간: 관리자 콘솔 - 승인대기 탭, 신청자 목록과 승인/거절 버튼]**
> *설명: 신청자 이름·이메일·가입 경로(이메일/카카오)가 나열되고, 각 항목 옆에 승인/거절 버튼이 있는 화면.*

3. 신청자 정보를 확인한 뒤 **승인**을 누르면 즉시 정회원으로 전환됩니다.
4. 여러 명을 한 번에 처리해야 한다면 체크박스로 다중 선택 후 일괄 승인도 가능합니다.
5. 승인이 완료되는 순간, 대기 중이던 회원의 화면은 별도 새로고침 없이 캘린더로 자동 전환됩니다.

## 3. 개발자 비하인드: OAuth는 왜 페이지를 통째로 떠났다가 돌아오는가

이메일 회원가입은 한 화면 안에서 폼을 채우고 제출하면 끝나는 단순한 흐름입니다. 반면 카카오 로그인은 `supabase.auth.signInWithOAuth({ provider: 'kakao' })`를 호출하는 순간, 브라우저가 우리 사이트를 완전히 떠나 카카오 로그인 페이지로 이동합니다. 사용자가 카카오에서 인증을 마치면 다시 우리 사이트로 돌아오는데, 이 과정에서 **리액트 메모리에 있던 모든 상태(state)는 사라집니다.** 페이지 자체가 새로 로드되는 것이니까요.

그래서 "가입 신청 전에 사용자가 골라둔 조직"처럼 OAuth 이동 전후로 유지해야 하는 값은, 리액트 state가 아니라 `localStorage`처럼 페이지 새로고침에도 살아남는 곳에 저장해야 합니다.

```typescript
// src/pages/AuthPage.tsx
async function handleKakaoOAuth() {
  if (wizChoice === 'join' && selectedTenantId) {
    localStorage.setItem('vs_pending_social', JSON.stringify({ tenantId: selectedTenantId, tenantRoleId: null }))
  } else {
    localStorage.setItem('vs_pending_mode', 'start-service')
  }
  const err = await signInWithKakao()  // 여기서 브라우저가 카카오로 이동
  ...
}
```

카카오 인증을 마치고 우리 사이트로 돌아오면, `TenantContext`가 이 `localStorage` 값을 읽어 "아, 이 사용자는 이 조직에 가입 신청을 하려던 참이었구나"를 파악하고 `tenant_members` 레코드를 자동으로 만듭니다. 리액트 컴포넌트 상태는 사라졌지만, 브라우저에 남겨둔 흔적으로 흐름을 이어 붙이는 방식입니다.

가입 직후 전화번호를 함께 받는 이메일 가입 흐름은 조금 더 직접적입니다. 회원가입 폼 제출 시 `raw_user_meta_data`에 전화번호를 함께 실어 보내고, DB 트리거(`handle_new_user`)가 `auth.users`에 새 계정이 생기는 순간 이 메타데이터를 읽어 `profiles.phone`에 그대로 저장합니다. 카카오처럼 메타데이터로 못 보내는 값(전화번호는 카카오 로그인 시점엔 아직 안 받은 정보)은, 로그인 이후 별도 화면에서 입력받아 `sessionStorage`에 잠깐 담아뒀다가 프로필 로드 시점에 업데이트하는 우회 경로를 씁니다.

## 4. Troubleshooting: 카카오로 가입했는데 조직 선택도 없이 승인 대기 화면부터 떴다

### 🔴 문제 상황

카카오 로그인 버튼을 회원가입 탭에 처음 붙였을 때, 로그인 탭의 "카카오로 로그인" 버튼과 똑같은 핸들러를 그대로 재사용했습니다. 이 핸들러는 곧바로 `signInWithKakao()`를 호출해 카카오로 이동시키는 게 전부였습니다.

로그인이라면 이게 맞습니다. 이미 가입된 계정이 다시 들어오는 것뿐이니까요. 하지만 **회원가입**은 다릅니다. "어느 조직에 가입 신청할 것인가"를 먼저 물어봐야 하는데, 이 질문 자체를 건너뛰고 바로 카카오로 넘겨버린 것이었습니다. 그 결과 사용자는 카카오 인증만 마치고 돌아와서, 아무 조직에도 소속되지 않은 채 승인 대기 화면(그마저도 대기할 조직이 없는 상태)을 마주하게 됐습니다.

### 🟢 해결 과정

핵심은 "회원가입 버튼 클릭"과 "실제 OAuth 리다이렉트"를 한 동작이 아니라 **두 단계로 분리**하는 것이었습니다.

```typescript
async function handleKakao() {
  if (tab === 'login') {
    // 로그인은 곧바로 OAuth
    const err = await signInWithKakao()
    ...
  } else {
    // 회원가입은 조직 선택 위자드부터 먼저 보여준다
    setKakaoWizMode(true)
    setWizChoice('join')
    setJoinStep('choice')
    setWizOpen(true)   // OAuth 호출은 아직 하지 않음
  }
}
```

회원가입 탭에서 카카오 버튼을 누르면 곧장 카카오로 가는 대신, 기존 이메일 가입 흐름에 이미 있던 **조직 선택 위자드**를 먼저 띄우도록 바꿨습니다. 사용자가 조직을 고르고 "카카오로 가입 신청하기" 버튼을 눌러야 비로소 `handleKakaoOAuth()`가 호출되며 앞서 본 것처럼 선택 내용을 `localStorage`에 저장한 뒤 카카오로 이동합니다.

> **📸 [이미지 삽입 구간: 수정 후 흐름 — 카카오 버튼 클릭 시 조직 선택 위자드가 먼저 뜨는 화면]**
> *설명: "회원가입" 탭에서 카카오 버튼을 누르면 카카오 사이트로 즉시 이동하는 대신, "가입할 조직을 선택하세요" 위자드 화면이 먼저 뜨고 하단에 "카카오로 가입 신청하기" 버튼이 나타나는 화면.*

기존에 이메일 가입용으로 만들어둔 위자드 컴포넌트를 그대로 재사용하되, "이 위자드가 지금 카카오 흐름의 일부로 열린 것인지"를 나타내는 `kakaoWizMode` 플래그 하나를 추가해 마지막 버튼의 동작(`handleSignUpOnly` 호출 vs `handleKakaoOAuth` 호출)만 분기하는 방식으로 풀었습니다. 화면 대부분을 새로 만들지 않고, "가입 신청 확정" 지점만 카카오용으로 갈아 끼운 셈입니다.
