# [제 7편] 새로고침 없이 — Realtime 구독과 조용히 새는 비용

여러 관리자가 동시에 같은 달력을 보고 있는 상황을 떠올려보겠습니다. 한 명이 셀 하나를 클릭해 배정을 추가하는 순간, 다른 관리자 화면에도 그 변화가 새로고침 없이 바로 나타나야 합니다. 그렇지 않으면 두 사람이 같은 시간대에 서로 다른 사람을 배정해버리는 충돌이 생깁니다.

Supabase Realtime은 이 문제를 깔끔하게 풀어주는 도구지만, 잘못 쓰면 청구서에서 먼저 문제를 알아차리게 되는 기능이기도 합니다. 7편은 이 둘 사이에서 균형을 잡은 이야기입니다.

---

## 1. 실사용 시나리오: 왜 실시간이어야만 하는가

### 시나리오: 코트 2면짜리 테니스 동호회

1편에서 다뤘던 테니스클럽 코트 예약 시나리오를 다시 가져와 보겠습니다. 회원 40명이 같은 캘린더를 보고 코트를 예약하는 회원공유 모드입니다. 이 경우 "새로고침해야 남의 예약이 보이는" 구조라면 어떻게 될까요. 두 회원이 거의 동시에 같은 코트·같은 시간을 클릭해 예약 신청을 넣는 상황이 실제로 일어납니다. DB 제약조건이 최종적으로 중복은 막아주지만, 뒤늦게 넣은 사람은 "방금 클릭했는데 왜 에러가 나죠?"라는 당혹스러운 경험을 하게 됩니다.

Realtime 구독이 켜져 있으면, 누군가 예약을 넣는 순간 다른 회원 화면의 그 셀은 즉시 "예약됨"으로 바뀝니다. 애초에 이미 채워진 셀을 클릭할 일 자체가 줄어드는 것이죠.

### 반대로 실시간이 필요 없는 데이터도 있다

모든 테이블을 다 실시간으로 구독하면 좋을 것 같지만, 실제로는 그렇지 않습니다. 조직 설정값(`tenants.settings`)이나 역할 목록(`tenant_roles`)처럼 관리자가 아주 가끔 바꾸는 데이터까지 실시간 구독을 걸어두면, 정작 중요한 배정 데이터의 구독과 자원을 나눠 쓰게 됩니다. 그래서 구독 대상은 "자주 바뀌고, 여러 사용자가 동시에 보고 있으며, 즉시 반영이 실제로 의미 있는" 테이블로만 좁혔습니다. 배정(`assignments`), 슬롯 하이라이트(`slot_highlights`), 날짜 예외(`date_overrides`) 이 세 테이블이 여기 해당합니다.

---

## 2. 관리자 조작 가이드: 사용자에게는 보이지 않는 기능

이 기능은 관리자가 따로 켜고 끄는 설정이 아닙니다. 회원공유·회원개별 모드에서는 항상 켜져 있고, 화면에 아무 표시도 없이 조용히 동작합니다. 그래서 "가이드"랄 게 따로 없지만, 사용자 입장에서 체감할 수 있는 지점은 있습니다.

> **📸 [이미지 삽입 구간: 두 브라우저 창을 나란히 띄운 스크린샷]**
> *설명: 왼쪽 창에서 셀을 클릭해 배정을 추가하는 순간, 오른쪽 창(다른 계정으로 로그인)의 같은 셀에 이름이 실시간으로 나타나는 화면.*

같은 조직의 캘린더를 두 개의 브라우저 창(또는 두 명의 로그인 세션)으로 열어두면, 한쪽에서 변경한 내용이 다른 쪽 화면에 별도 조작 없이 곧바로 반영됩니다. 배정 추가·삭제뿐 아니라 날짜 오버라이드(휴관일 지정 등)도 동일하게 실시간으로 퍼집니다.

## 3. 개발자 비하인드: 구독 하나에 숨어있는 세 가지 함정

Realtime 구독을 걸 때 신경 써야 할 지점을 정리해 별도 문서(`docs/realtime-subscription-status.md`)로 관리하고 있습니다. 핵심은 세 가지입니다.

**첫째, 언마운트 시 구독 해제.** 컴포넌트가 사라질 때 구독을 정리하지 않으면 동시 접속자 수만큼 좀비 연결이 쌓입니다.

```typescript
useEffect(() => {
  const channel = supabase.channel('...').on(...).subscribe()
  return () => { supabase.removeChannel(channel) }  // 필수
}, [tenantId])
```

**둘째, `tenant_id` 필터.** 테이블 전체를 구독하면 다른 조직의 변경 이벤트까지 전부 내 브라우저로 날아옵니다. 조직이 100개면 내 배정 하나 바뀔 때마다 100개 조직 몫의 이벤트를 다 받게 되는 셈이라, 필터 없이는 메시지 수가 조직 수에 비례해 폭증합니다.

```typescript
.on('postgres_changes', {
  event: 'INSERT', schema: 'public', table: 'slot_highlights',
  filter: `tenant_id=eq.${tenantId}`,
}, handler)
```

**셋째, INSERT/UPDATE는 몰라도 DELETE 필터는 그냥 걸면 안 됩니다.** 이 셋째 함정이 실제로 한 번 제대로 발목을 잡았던 지점이라, 다음 장에서 따로 다루겠습니다.

## 4. Troubleshooting: 분명히 필터를 걸었는데 삭제 이벤트가 하나도 안 온다

### 🔴 문제 상황

`slot_highlights`(셀 강조 표시) 기능에 실시간 구독을 붙이면서, INSERT와 마찬가지로 DELETE에도 `tenant_id` 필터를 그대로 걸었습니다.

```typescript
.on('postgres_changes', {
  event: 'DELETE', schema: 'public', table: 'slot_highlights',
  filter: `tenant_id=eq.${tenantId}`,
}, handler)
```

INSERT는 잘 오는데, 다른 사용자가 하이라이트를 지우면 내 화면에서는 아무 일도 일어나지 않았습니다. 필터 조건 자체는 문법적으로 문제가 없어 보였는데, 이벤트가 서버에서 아예 안 오는 것처럼 보였습니다.

> **📸 [이미지 삽입 구간: 브라우저 콘솔 - Realtime 이벤트 로그]**
> *설명: 왼쪽 창에서 하이라이트를 지웠는데도, 오른쪽 창의 콘솔에는 INSERT 이벤트 로그만 찍히고 DELETE 이벤트는 전혀 찍히지 않는 개발자도구 스크린샷.*

### 🟢 원인과 해결

PostgreSQL의 `REPLICA IDENTITY` 기본값(`DEFAULT`)에서는, DELETE 이벤트의 `payload.old`에 **기본키(PK)만** 담깁니다. `tenant_id`는 PK가 아니니 `payload.old`에 아예 없고, Supabase Realtime 서버는 필터에 쓰인 컬럼이 페이로드에 없으면 그 이벤트 자체를 클라이언트에 전달하지 않습니다. 필터 문법이 틀린 게 아니라, 애초에 서버가 판단할 재료가 없어서 조용히 버려지고 있었던 것입니다.

수정은 두 단계로 진행했습니다.

1. **우선 필터를 걷어내는 임시 조치.** DELETE 구독에서 `tenant_id` 필터를 빼고, 대신 클라이언트가 이미 들고 있는 하이라이트 상태에서 `id`로 직접 매칭해 삭제 처리하도록 바꿨습니다. 필터 없이 전체 이벤트를 받되, 내 조직 것만 걸러내는 책임을 서버에서 클라이언트로 잠시 옮긴 셈입니다.

2. **근본 해결.** `slot_highlights` 테이블에 `REPLICA IDENTITY FULL`을 적용했습니다. 이렇게 하면 DELETE 이벤트의 `payload.old`에 삭제된 행의 **모든 컬럼**이 담기고, `tenant_id` 필터를 다시 걸어도 서버가 정상적으로 판단할 수 있습니다.

```sql
ALTER TABLE slot_highlights REPLICA IDENTITY FULL;
```

이 경험 이후로 "DELETE 이벤트에 특정 컬럼으로 필터를 걸어야 하는 테이블은 반드시 `REPLICA IDENTITY FULL`을 함께 적용한다"를 프로젝트 규칙으로 문서에 못박아뒀습니다. INSERT·UPDATE 이벤트는 페이로드에 변경 후 전체 행이 실리기 때문에 이 문제가 아예 존재하지 않는데, 유독 DELETE만 이렇게 동작한다는 걸 모르면 "필터를 걸었는데 왜 안 오지?"라는 질문에서 한참을 헤매게 됩니다.

---

## 5. 7편을 마치며

실시간 동기화라는 매끄러운 사용자 경험 뒤에서, DELETE 이벤트 하나가 PostgreSQL의 복제 정체성(replica identity) 설정에 따라 완전히 다르게 동작한다는 걸 확인한 이야기였습니다. 겉보기엔 똑같은 `postgres_changes` 구독인데, 이벤트 종류에 따라 규칙이 다르다는 걸 실제로 겪어보지 않았다면 놓치기 쉬운 지점이었습니다.
