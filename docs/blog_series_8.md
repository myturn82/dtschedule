# [제 8편] 엑셀을 미워하지 않기로 했다 — 붙여넣기 모드와 4종 내보내기

스프레드시트의 해방을 표방하며 시작한 서비스인데, 정작 만들다 보니 엑셀과 완전히 결별할 수는 없었습니다. 수십 년간 엑셀로 스케줄을 짜온 관리자들에게 "이제 마우스로 한 칸씩 클릭하세요"라고 하면 오히려 반발이 생깁니다. 그래서 8편은 엑셀을 적으로 돌리는 대신, 엑셀의 익숙한 조작감을 캘린더 안으로 가져온 이야기입니다.

---

## 1. 실사용 시나리오: 손에 익은 그 느낌 그대로

### 시나리오: 반복 패턴이 많은 대형 카페 알바 스케줄

알바생 15명이 3교대로 돌아가는 카페를 예로 들어보겠습니다. 이런 조직은 배정 패턴이 반복적입니다. "이번 주 오픈조 3명"을 정했으면, 다음 주도 크게 다르지 않은 경우가 많습니다. 셀 하나하나를 클릭해 이름을 입력하는 대신, 이미 짜둔 한 주를 통째로 복사해서 다음 주에 붙여넣을 수 있다면 관리 시간이 확 줄어듭니다.

그래서 만든 게 **엑셀모드**입니다. 캘린더 화면에서 엑셀처럼 셀을 드래그해 범위를 선택하고, `Ctrl+C`/`Ctrl+V`로 복사·붙여넣기가 됩니다. PC에서는 Shift+클릭으로 범위를 확장하고, 모바일에서는 터치로 셀을 눌러 범위를 지정합니다.

역할이 나뉜 조직(3편 참고)이라면 여기에 규칙이 하나 더 붙습니다. "커트팀" 칸에서 복사한 내용을 "펌·염색팀" 칸에 붙여넣으면 안 되겠죠. 그래서 붙여넣기는 항상 원본 셀과 같은 역할 칸에만 적용되도록 소속 규칙을 검사합니다.

> **📸 [이미지 삽입 구간: 오픈조 3칸 복사 → 다음 주 같은 칸에 붙여넣기 직후 화면]**
> *설명: 월요일 오픈조 3명이 선택된 상태(파란 테두리)에서, 붙여넣기 직후 다음 주 월요일 칸에 동일한 3명이 채워지는 모습을 전/후로 비교한 스크린샷.*

### 시나리오: 인쇄해서 벽에 붙여야 하는 현장

반대로 디지털 화면이 아니라 종이가 필요한 순간도 있습니다. 요양보호사들이 현장에 나가면 스마트폰을 못 볼 때가 많은 조직도 있고, 원장님이 이번 달 스케줄을 인쇄해서 직원 휴게실 게시판에 붙여야 하는 경우도 있습니다. 이런 요구에 맞춰 **엑셀·CSV·워드(DOCX)·PDF** 네 가지 포맷으로 한 달 치 스케줄을 내보내는 기능을 붙였습니다.

## 2. 관리자 조작 가이드: 드래그 한 번으로 한 주 복사하기

> **📸 [이미지 삽입 구간: 엑셀모드 활성화 후 셀 범위 드래그 선택 화면]**
> *설명: 월요일~금요일 오픈조 3칸이 파란 테두리로 선택된 상태에서, 우클릭 메뉴 또는 단축키 안내가 뜨는 화면.*

1. 캘린더 화면 상단의 **엑셀모드** 토글을 켭니다.
2. 복사하고 싶은 범위를 드래그(PC) 또는 터치 드래그(모바일)로 선택합니다.
3. `Ctrl+C`(또는 복사 버튼)를 누릅니다.
4. 붙여넣을 시작 셀을 클릭한 뒤 `Ctrl+V`를 누르면, 선택했던 범위 크기 그대로 데이터가 복제됩니다.
5. 마음에 들지 않으면 `Ctrl+Z`로 되돌릴 수 있습니다.

> **📸 [이미지 삽입 구간: 내보내기 버튼 클릭 시 포맷 선택 드롭다운]**
> *설명: "엑셀(.xlsx) / CSV / 워드(.docx) / PDF" 네 개 버튼이 나열된 내보내기 메뉴.*

6. 화면 상단 **내보내기** 버튼을 누르면 네 가지 포맷 중 하나를 고를 수 있습니다.
7. 엑셀·CSV는 데이터 재가공(필터링, 정렬)에, 워드·PDF는 그대로 인쇄하거나 공유하기에 적합합니다.

## 3. 개발자 비하인드: 셀 선택은 상태 하나로, PC와 모바일을 함께 다루기

엑셀모드의 핵심은 "지금 어떤 셀이 선택돼 있는가"를 나타내는 상태 하나입니다. 이 상태는 `anchor`(선택을 시작한 셀)와 `cursor`(현재 가리키는 셀) 두 좌표만으로 표현합니다.

```typescript
// src/utils/excelSelection.ts
export function nextCellSelection(
  prev: { anchor: CellPos; cursor: CellPos } | null,
  pos: CellPos,
  forceExtend: boolean
): { anchor: CellPos; cursor: CellPos } {
  if (forceExtend && prev) return { anchor: prev.anchor, cursor: pos }
  if (prev && !isSameCell(prev.anchor, prev.cursor)) return { anchor: pos, cursor: pos }
  if (prev) return { anchor: prev.anchor, cursor: pos }
  return { anchor: pos, cursor: pos }
}
```

흥미로운 지점은 이 함수 하나로 PC의 "클릭 vs Shift+클릭"과 모바일의 "탭 vs 드래그"를 동시에 처리한다는 것입니다. `forceExtend`가 PC의 Shift 키 여부를 나타내는데, 이게 켜져 있으면 무조건 `cursor`만 갱신해 범위를 넓힙니다. 꺼져 있을 때는 조금 더 섬세한 규칙이 필요합니다. 이전 선택이 이미 범위(anchor와 cursor가 다른 상태)였다면 새 클릭은 "새로운 단일 셀 선택 시작"으로, 아직 단일 셀(anchor === cursor)이었다면 새 클릭은 "그 셀을 기준으로 범위 확장"으로 해석합니다.

이 규칙 덕분에 마우스로 셀을 한 번 클릭하고, 다른 셀을 다시 클릭하면 그 사이 범위가 선택되는 "엑셀에서 흔히 쓰는 동작"이 자연스럽게 재현됩니다. 모바일에서는 Shift 키가 없으니 `forceExtend`를 항상 false로 두고, 터치 드래그의 시작·끝 좌표를 그대로 `anchor`/`cursor`에 매핑하는 방식으로 같은 함수를 재사용합니다.

내보내기 쪽은 완전히 다른 라이브러리 네 개를 조합합니다. 엑셀은 `exceljs`, 워드는 `docx`, PDF는 `html2canvas`로 캘린더 DOM을 이미지로 캡처한 뒤 `jsPDF`로 감싸는 방식입니다. 네 포맷이 서로 다른 렌더링 파이프라인을 쓰다 보니, 공통 로직(주차 계산, 공휴일 판정, 마스킹 규칙)은 유틸 함수로 뽑아 각 포맷 생성 함수가 공유하도록 만들었습니다.

## 4. Troubleshooting: 마스킹 함수인데 마스킹을 안 하고 있었다

### 🔴 문제 상황

고객 연락처를 다루는 예약제 업종(부동산, 오마카세 식당, 사진 스튜디오 등)에서는 일반 회원에게 다른 고객의 전화번호 전체가 그대로 노출되면 안 됩니다. 그래서 `maskPhone`/`maskEmail`/`maskName`이라는 마스킹 함수를 처음부터 만들어뒀습니다. 문제는 이 함수들의 실제 구현이었습니다.

```typescript
// 수정 전
export function maskPhone(v: string): string { return fmtPhone(v) }  // 그냥 포맷만 하고 그대로 반환
export function maskEmail(v: string): string { return v }             // 원본 그대로
export function maskName(v: string):  string { return v }             // 원본 그대로
```

함수 이름은 "마스킹"인데 실제로는 아무것도 가리지 않는 **패스스루(pass-through)** 상태였습니다. 화면 컴포넌트들은 "관리자가 아니면 `maskPhone()`을 거쳐서 보여준다"는 규칙을 성실히 지키고 있었는데, 정작 그 함수가 원본을 그대로 돌려주고 있었으니 규칙 자체가 무의미했던 셈입니다. 엑셀·CSV·워드·PDF 내보내기 파일도 마찬가지로, 관리자 권한 없이 다운로드해도 고객 전화번호가 그대로 찍혀 나왔습니다.

> **📸 [이미지 삽입 구간: 수정 전/후 내보낸 엑셀 파일 비교]**
> *설명: 왼쪽은 일반 회원이 내려받은 엑셀에 고객 전화번호가 "010-1234-5678" 그대로 찍힌 모습, 오른쪽은 수정 후 "010-****-5678"로 마스킹된 모습을 나란히 비교한 스크린샷.*

### 🟢 해결 과정

먼저 세 함수를 실제로 마스킹하도록 새로 구현했습니다.

```typescript
// 수정 후 — src/lib/format.ts
export function maskPhone(value: string | null | undefined): string {
  const formatted = fmtPhone(value)
  if (!formatted) return ''
  const parts = formatted.split('-')
  if (parts.length !== 3) return formatted
  return `${parts[0]}-****-${parts[2]}`   // 010-****-5678
}
```

이름과 이메일도 각각 "첫 글자와 마지막 글자만 남기고 중간을 별표 처리", "이메일 아이디 앞 2글자만 남기고 도메인은 그대로" 방식으로 구현했습니다.

더 중요한 작업은 그다음이었습니다. 함수를 고친 뒤, **이 함수를 호출해야 하는데 빠뜨린 지점이 없는지** 코드베이스 전체를 훑었습니다. 월간 뷰(`TimeSlotCell`), 일간 뷰(`DayView`), 배정 등록 모달(`SlotEditModal`), 그리고 엑셀·CSV·워드·PDF 내보내기(`exportSchedule.ts`)까지 총 7개 파일을 손봐야 했습니다. 흥미롭게도 주간 뷰(`WeekGrid`)는 애초에 고객 정보를 검색 매칭 용도로만 내부적으로 쓰고 화면에 직접 노출하지 않아서 수정 대상에서 제외됐는데, 이것도 "화면마다 노출 경로가 다르니 하나하나 확인해야 한다"는 걸 보여주는 사례였습니다.

이 사건은 "함수 이름이 곧 함수의 동작을 보장하지 않는다"는, 당연하지만 쉽게 잊히는 교훈을 남겼습니다. `maskPhone`이라는 이름과 시그니처만 보고 안심한 채 여러 화면에 흩어 심어뒀는데, 정작 구현체를 열어보지 않으면 아무도 그게 무의미한 패스스루였다는 걸 알아채기 어려웠습니다. 이후로는 개인정보 마스킹처럼 보안에 직결되는 유틸 함수는 반드시 실제 출력값을 테스트로 검증하는 걸 원칙으로 삼았습니다.

## 5. 8편을 마치며

익숙한 엑셀 조작감을 캘린더 안에 재현한 이야기, 그리고 이름은 멀쩡한데 실제로는 아무 일도 하지 않던 마스킹 함수를 발견하고 고친 이야기를 다뤘습니다. 겉으로 드러난 기능은 멀쩡해 보여도, 그 안의 구현이 이름값을 하고 있는지는 따로 확인해야 한다는 걸 다시 확인한 편이었습니다.
