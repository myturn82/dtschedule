# [제 11편, 완결] 브라우저가 알아서 해줄 줄 알았다 — 다크모드와 디자인 시스템

시리즈의 마지막 편입니다. 1편부터 10편까지 기능 하나하나를 쌓아 올렸다면, 11편은 그 모든 화면을 관통하는 "보이는 방식"에 관한 이야기입니다. 다크모드는 요즘 웹서비스라면 거의 기본 옵션처럼 취급받지만, 막상 만들어보면 "그냥 색만 반전시키면 되는 거 아닌가"라는 생각이 얼마나 순진했는지 깨닫게 됩니다.

## 1. 실사용 시나리오: 새벽에 스케줄을 확인하는 사람들

### 시나리오: 야간 배송 기사들의 배차 확인

1편에서 다룬 퀵서비스·택배 배차 시나리오(M)를 다시 가져와 보겠습니다. 새벽 배송을 준비하는 기사님들은 어두운 방이나 운전석에서 휴대폰으로 오늘 배차를 확인합니다. 흰 배경에 검은 글씨로 가득한 화면을 새벽에 마주하면 눈이 부십니다. 다크모드는 단순한 취향 옵션이 아니라, 실제 사용 환경을 고려한 필수 기능이었습니다.

### 아이콘은 이모지로 통일한 이유

색상 얘기가 나온 김에 아이콘 원칙도 짚어볼 만합니다. 이 서비스는 별도 아이콘 폰트나 SVG 아이콘 세트를 쓰지 않고, **시스템 기본 이모지**를 아이콘으로 씁니다. 이유는 단순합니다. 아이콘 폰트를 추가로 로드하지 않아도 되고, Windows(Segoe UI Emoji)든 macOS(Apple Color Emoji)든 각 OS가 이미 가진 폰트를 그대로 활용하니 번들 크기가 늘지 않습니다. 대신 "이모지는 색상을 CSS로 바꿀 수 없다"는 제약을 받아들여야 했고, 그래서 잠금 아이콘처럼 세밀한 크기·색 제어가 필요한 곳만 예외적으로 SVG를 씁니다.

> **📸 [이미지 삽입 구간: 같은 캘린더 화면의 라이트모드/다크모드 비교]**
> *설명: 동일한 배차 캘린더를 왼쪽은 라이트모드(밝은 배경), 오른쪽은 다크모드(어두운 배경)로 나란히 배치한 스크린샷. 이모지 아이콘(🔒, 🌙 등)이 두 모드에서 동일하게 보이는 것도 함께 확인 가능.*

## 2. 관리자 조작 가이드: 다크모드 전환

> **📸 [이미지 삽입 구간: 프로필 메뉴 - 다크모드 토글]**
> *설명: 우측 상단 프로필 메뉴에서 해/달 아이콘과 함께 "다크모드" 항목을 클릭하면 화면 전체가 즉시 어두운 팔레트로 전환되는 화면.*

1. 우측 상단 프로필 메뉴에서 **다크모드** 항목을 클릭합니다.
2. 화면 전체(캘린더, 관리자 콘솔, 모달까지) 색상이 즉시 전환됩니다.
3. 별도로 설정하지 않으면 기기의 OS 설정(라이트/다크)을 따라갑니다. 한 번이라도 수동으로 전환하면, 그 이후로는 OS 설정이 바뀌어도 사용자가 고른 값을 그대로 유지합니다.

## 3. 개발자 비하인드: 색상 하나 바꾸면 전체가 바뀌는 구조

다크모드 전환은 컴포넌트마다 조건문을 넣는 방식이 아니라, **CSS 커스텀 프로퍼티(변수) 하나를 스위칭**하는 방식으로 만들었습니다.

```css
/* src/index.css */
:root {
  --color-bg: #F4F1EA;
  --color-surface: #FFFFFF;
  --color-text-primary: #14171C;
  ...
}
.dark {
  --color-bg: #0c0e14;
  --color-surface: #13161e;
  --color-text-primary: #f1f5f9;
  ...
}
```

`<html>` 태그에 `dark` 클래스를 붙였다 뗐다 하는 것만으로, `var(--color-bg)`를 참조하는 모든 컴포넌트의 색이 한꺼번에 바뀝니다. 컴포넌트 코드에는 `dark:` 같은 조건부 클래스를 일일이 붙일 필요가 없습니다. Tailwind CSS 4의 `@variant` 문법으로 `.dark` 클래스 기반 다크 변형을 한 줄로 정의해뒀고, 필요한 곳에서만 `dark:` 유틸리티를 보조적으로 씁니다.

```typescript
// src/contexts/DarkModeContext.tsx
useEffect(() => {
  if (isDark) document.documentElement.classList.add('dark')
  else document.documentElement.classList.remove('dark')
  localStorage.setItem('darkMode', String(isDark))
}, [isDark])
```

이 구조 덕분에 새 컴포넌트를 만들 때 다크모드를 "따로 신경 쓸 필요가 없는" 상태가 됩니다. `var(--color-text-primary)`처럼 의미 기반 변수 이름만 쓰면, 라이트든 다크든 알아서 맞는 색이 나옵니다.

## 4. Troubleshooting: `color-scheme`을 선언했는데 왜 여전히 흰 배경일까

### 🔴 문제 상황

CSS 변수로 대부분의 다크모드는 매끄럽게 처리됐는데, 딱 하나 말을 안 듣는 요소가 있었습니다. 관리자 콘솔의 조직 선택 `<select>`였습니다. 닫혀 있는 select 박스 자체는 다크 팔레트로 잘 나오는데, 클릭해서 **펼쳐지는 옵션 목록(팝업)**만은 항상 흰 배경에 밝은 글자색이 겹쳐 텍스트가 거의 안 보이는 상태였습니다.

> **📸 [이미지 삽입 구간: 다크모드에서 흰 배경으로 뜨는 select 팝업 — 실제 발견 당시 스크린샷]**
> *설명: 어두운 배경의 관리자 화면 위에, 조직 선택 드롭다운을 열었을 때만 흰색 팝업이 튀어나오고 그 안의 조직명 텍스트가 잘 안 보이는 실제 버그 재현 화면.*

원인은 명확했습니다. `<select>`의 옵션 팝업은 브라우저가 직접 그리는 네이티브 UI라서, 우리가 CSS 변수로 준 배경색이 전혀 적용되지 않습니다. 이걸 제어하는 표준 CSS 속성이 `color-scheme`인데, `:root`에는 `light`를, `.dark`에는 `dark`를 선언해뒀습니다.

```css
:root { color-scheme: light; }
.dark { color-scheme: dark; }
```

이론적으로는 이 두 줄이면 브라우저가 알아서 네이티브 select 팝업을 다크 팔레트로 그려줘야 합니다. 실제 배포된 CSS 번들에도 이 선언이 정확히 들어가 있는 걸 확인했고, 하드 리프레시까지 해봤지만 여전히 흰 배경이었습니다.

### 🟢 해결 과정

`color-scheme`은 스펙상으로는 표준이지만, 실제로는 브라우저·OS 조합에 따라 `<select>` 팝업 렌더링에 얼마나 반영하는지가 제각각입니다. 표준을 따랐다고 해서 모든 환경에서 기대한 대로 그려진다는 보장이 없다는 걸 실제로 겪고 나서야 알았습니다.

그래서 접근 방식을 바꿨습니다. 브라우저의 "알아서 그려주는" 동작에 기대는 대신, `<option>` 요소에 배경색과 글자색을 **직접 지정**하는 훨씬 오래되고 확실한 방법을 썼습니다.

```css
/* src/index.css */
.dark select,
.dark select option {
  background-color: var(--color-surface);
  color: var(--color-text-primary);
}
```

`background-color`와 `color`는 `<option>` 요소에 직접 지정하면 대부분의 브라우저가 팝업 렌더링에 실제로 반영합니다. `color-scheme`보다 훨씬 오래전부터 지원되어온 속성이라 호환성 편차도 적습니다. 결과적으로 `color-scheme` 선언은 스크롤바나 다른 네이티브 UI를 위해 그대로 남겨두고, select 팝업 문제는 이 명시적 스타일로 별도 해결했습니다.

> **📸 [이미지 삽입 구간: 수정 후 — 다크모드에서 정상적으로 어두운 배경으로 뜨는 select 팝업]**
> *설명: 같은 조직 선택 드롭다운을 열었을 때, 이번엔 팝업 배경도 어둡게 렌더링되고 조직명 텍스트가 또렷하게 읽히는 화면. 바로 위 버그 스크린샷과 나란히 배치하면 전후 비교 효과.*

이 경험에서 얻은 교훈은 단순합니다. 표준 스펙이 있다고 해서 그게 곧 "모든 브라우저에서 기대한 대로 동작한다"는 보장은 아니라는 것입니다. 특히 `<select>`, `<input type="date">`처럼 브라우저가 직접 렌더링을 담당하는 네이티브 폼 컨트롤은, CSS 변수 기반 테마 시스템의 영향력이 미치지 않는 사각지대라는 걸 이번에 실감했습니다. 새 네이티브 컨트롤을 쓸 때는 다크모드에서 실제로 열어보고 확인하는 습관이 생겼습니다.
